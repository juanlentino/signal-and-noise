## 2. Plugin v4.5.0

Three workstreams. Two refinement, one net-new.

### Workstream 1 — `_deprecated_function()` annotations on legacy REST handlers (~30 LOC)

**What:** the 4 REST handlers documented as `@deprecated 2.5.0` since v2.5.0 (16 versions ago) still ship without runtime deprecation signaling.

- `snt_ai_meta_desc_rest_handler` → `/ai/generate-meta-description`
- `snt_ai_excerpt_rest_handler` → `/ai/generate-excerpt`
- `snt_ai_og_card_title_rest_handler` → `/ai/generate-og-card-title`
- `snt_desktop_cmd_handler` → `/cmd/<action>`

**Change:** add `_deprecated_function( __FUNCTION__, '2.5.0', 'wp-abilities/v1/abilities/<slug>/run' )` to each handler. Fires `WP_DEBUG_LOG` notices when the legacy route is hit.

**Why this matters:** signal-only. Doesn't remove anything. But anyone running `WP_DEBUG_LOG` gets an audit trail of which legacy routes are still being exercised. That tells us when it's safe to actually remove them in v5.

### Workstream 2 — JS-client flip: ability-first, REST-fallback (~100 LOC across 3 files)

**What:** `assets/ai-meta-description.js`, `ai-excerpt.js`, `ai-og-card-title.js` currently try the legacy REST route first and fall back to the Abilities API path. Flip the order: ability path first, REST fallback only if ability returns 404.

**Why:** new installs running WP 7.0+ (where Abilities API ships) never need to hit the legacy REST routes after this. Once `_deprecated_function()` logs (Workstream 1) show zero traffic for a few months, v5.0.0 can safely remove them.

**`assets/command-palette.js` is OUT of scope.** Consumed by the external `WordPress/desktop-mode` plugin (not under our control). Stays REST-first until WordPress/desktop-mode migrates to abilities upstream.

### Workstream 3 — Block Migration tool (NET-NEW, ~600 LOC + ~120 LOC tests)

Directly addresses the SQL-miss we hit this session (PA-01 content fix). Generalizes for all future content migrations.

**File structure (5 new files, mirrors the v4.3.0 pattern-adoption naming):**

```
inc/block-migrations-detect.php    ← scan + walker + transient cache
inc/block-migrations-suggest.php   ← deterministic before/after preview
inc/block-migrations-apply.php     ← block-tree mutation + wp_update_post
inc/block-migrations-admin.php     ← UI renderer (Tools tab section)
inc/abilities-block-migrations.php ← 4 Ability registrations
```

Plus tests: `tests/block-migrations-detect.php`, `tests/block-migrations-suggest.php`, `tests/block-migrations-apply.php`.

**Migration registry — v4.5.0 ships ONE migration:**

```php
const SNT_BLOCK_MIGRATIONS_VALID_TYPES = array( 'heading-hierarchy-skip' );
```

Hardcoded list. YAGNI — registry abstraction deferred until 2nd migration arrives.

**Heading-skip detection logic:**

Walks all `post_type=post` `post_status=publish` posts. For each post:

1. `parse_blocks($post->post_content)` → block tree
2. Walk top-level + innerBlocks recursively
3. For each `core/heading` node where `attrs.level === 3`: check the *preceding* heading context in the same post — if NO `h2` exists between the post's `<h1>` (the post title — implicit, not in `post_content`) and this `<h3>`, it's a skip candidate
4. Fingerprint = `md5(serialize_block($node))`
5. Skip if `"heading-hierarchy-skip:{$fingerprint}"` is in `_snt_block_migrations_dismissed` post meta
6. Append `{post_id, migration_type, block_fingerprint, block_path, post_title, permalink, current_level, target_level}` to candidates

Cached in per-user transient `snt_block_migrations_candidates_<user_id>` at `HOUR_IN_SECONDS`. Same TTL convention as pattern-adoption.

**Detection nuance:** only flags h3s with NO preceding h2 in their post. If hierarchy is h1 → h2 → h2 → h3 (valid), we don't migrate.

**Suggest impl (deterministic — no AI, mirrors `snt_pattern_adoption_suggest_impl`):**

Input: `(post_id, block_fingerprint, migration_type)`.

1. Load post + parse_blocks
2. Walk for fingerprint match (recursive search — same pattern as `snt_pattern_adoption_find_block`)
3. If not found → `snt_block_migration_candidate_not_found` (404)
4. Branch on `migration_type` → `snt_block_migrations_build_heading_promotion($match_block)`
5. Return `{ok, suggestion_markup, fingerprint, post_id, migration_type, before_summary, after_summary}`

`snt_block_migrations_build_heading_promotion` is the actual transformation:

```php
function snt_block_migrations_build_heading_promotion( $heading_block ) {
    // Mutate level: unset (so serialize_block omits the JSON key — canonical for h2)
    unset( $heading_block['attrs']['level'] );

    // Mutate innerContent — what serialize_block actually uses
    foreach ( $heading_block['innerContent'] as $i => $content ) {
        if ( is_string( $content ) ) {
            $heading_block['innerContent'][ $i ] = preg_replace(
                array( '/<h3(\b[^>]*)>/', '/<\/h3>/' ),
                array( '<h2$1>', '</h2>' ),
                $content
            );
        }
    }

    // innerHTML mutation is informational only — serialize_block ignores it.
    // Update for consistency with downstream code that inspects parse_blocks output.
    if ( isset( $heading_block['innerHTML'] ) ) {
        $heading_block['innerHTML'] = preg_replace(
            array( '/<h3(\b[^>]*)>/', '/<\/h3>/' ),
            array( '<h2$1>', '</h2>' ),
            $heading_block['innerHTML']
        );
    }

    return serialize_block( $heading_block );
}
```

**Apply impl (mirrors `snt_ai_pattern_adoption_apply_impl` exactly):**

1. Permission check: `current_user_can( 'edit_post', $post_id )`
2. Post existence check
3. `parse_blocks($post->post_content)` → tree
4. Walk for fingerprint match, replace matching node with `parse_blocks($replacement_markup)[0]`
5. If not found → `snt_block_migration_conflict` (409)
6. `serialize_blocks($tree)` → new post_content
7. `wp_update_post( ['ID' => $post_id, 'post_content' => $new_content], true )`
8. Return `{ok, post_id, migration_type}`

**Admin UI surface (Tools tab section):**

Renders via `add_action( 'sn_admin_tools_tab', 'snt_block_migrations_render_section' )`. If the hook doesn't already exist in `admin-page.php`, add it alongside the existing 9 `sn_admin_*_tab` extension points (~5 LOC addition).

UI mirrors `snt_pattern_adoption_render_opportunities_section`:
- Heading + count pill + "Scan for block migrations" form button
- Collapsed `<details>` results table (Post / Migration type / Action columns)
- Per-row buttons: `[Suggest]` (data-attrs trigger shared `health-suggest-actions.js`) + `[Dismiss]`

**Abilities API:**

4 ability registrations under category **`tools`** (NOT `ai-generation` — these aren't AI). All with `meta.annotations.idempotent: true`; apply also gets `destructive: true`.

```

signal-noise/block-migrations-apply
signal-noise/block-migrations-dismiss
```

REST surface mirrors abilities:

```

POST /signal-noise/v1/tools/block-migrations-suggest
POST /signal-noise/v1/tools/block-migrations-apply

```

**Frontend JS — extend `health-suggest-actions.js` (~30 LOC additions):**

Add to `ABILITY_BY_CHECK` map:

```javascript

    suggest: 'block-migrations-suggest',
    apply:   'block-migrations-apply',
}
```

Apply uses the existing v4.0.3 Before/After modal. New render branch for `block-migration` finding type (~40 LOC in the modal helper).

**Safety properties:**

1. **Block-level fingerprint** — `md5(serialize_block($node))`. Any change to the block between scan and apply → fingerprint mismatch → 409.
2. **No bulk-apply** — per `feedback_health_suggest_apply_pattern` memory: *"Apply stays per-row — never batch-apply."*
3. **WP revisions preserved** — `wp_update_post` creates a revision automatically. Restore via WP's native UI if needed.
4. **Idempotent scans** — re-scanning after apply shows zero candidates.
5. **Dismiss per-finding** — `_snt_block_migrations_dismissed` post meta.

**Sequencing within v4.5.0:** Workstream 1 → Workstream 2 → Workstream 3. Total ~750 LOC + 120 LOC tests.

**What this UNLOCKS for v5:** future content migrations all use this infrastructure. No more bespoke SQL. v5.0.0's option-cleanup migration can run through this same registry pattern.

### Workstream 3 — Build-time contrast verification (net-new, ~120 LOC + ~25 assertions)

**New test file: `tests/contrast-baseline.php`**

WCAG 2.1 relative-luminance algorithm:

```php
function snt_test_relative_luminance( $hex ) {
    $hex = ltrim( $hex, '#' );
    $r = hexdec( substr( $hex, 0, 2 ) ) / 255;
    $g = hexdec( substr( $hex, 2, 2 ) ) / 255;
    $b = hexdec( substr( $hex, 4, 2 ) ) / 255;

    foreach ( array( 'r', 'g', 'b' ) as $c ) {
        $$c = ( $$c <= 0.03928 ) ? $$c / 12.92 : pow( ( $$c + 0.055 ) / 1.055, 2.4 );
    }
    return 0.2126 * $r + 0.7152 * $g + 0.0722 * $b;
}

function snt_test_contrast_ratio( $hex_a, $hex_b ) {
    $l1 = snt_test_relative_luminance( $hex_a );
    $l2 = snt_test_relative_luminance( $hex_b );
    $lighter = max( $l1, $l2 );
    $darker  = min( $l1, $l2 );
    return ( $lighter + 0.05 ) / ( $darker + 0.05 );
}
```

**Test assertions** (codifies ACCESSIBILITY.md §Current palette + §Watch thresholds):

**Helper addition to harness:**

```php
function ha_gte( $actual, $expected, $msg ) {
    global $pass, $fail;
    if ( $actual >= $expected ) {
        $pass++; echo "  PASS: $msg (got " . sprintf('%.2f', $actual) . ":1)\n";
    } else {
        $fail++; echo "  FAIL: $msg (expected ≥ $expected, got " . sprintf('%.2f', $actual) . ":1)\n";
    }
}
```

## 6. Rollback safety

**Per-tag rollback:**
- Plugin: `gh workflow run deploy.yml --repo juanlentino/signal-and-noise-tools --ref v4.4.5` reverts to current pre-v4.5.0

**Block Migration apply rollback (per-finding):**
- Every apply runs through `wp_update_post` → WP revision created automatically
- User can restore any post's prior content via wp-admin → Posts → Revisions
- Bulk recovery: WP database backup (Cloudways UI or `wp db export`)

**v10.0.0-scope.md is doc-only** — no rollback needed.

**Contrast baseline test rollback** — failed test = signal to either revert palette change OR update test (deliberate brand evolution). Normal git workflow.

## 8. Risks + mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Block Migration: heading detection edge cases (nested in groups/columns) | Medium | Recursive `innerBlocks` walker (verified pattern from pattern-adoption-detect.php) |
| Block Migration: parse_blocks → modify → serialize_blocks round-trip drift | Low | Pattern-adoption ships this exact round-trip in production at v4.3.0; tests verify equivalence |
| Block Migration: comments / shortcodes in post_content confuse parser | Low | parse_blocks is the WP-canonical parser; whatever the editor accepts, parse_blocks accepts |
| `_deprecated_function()` log spam (with WP_DEBUG_LOG=true) | Medium | Acceptable — the spam IS the signal. Production usually WP_DEBUG_LOG=false |
| JS-client flip causes regression in editor surfaces | Medium | UAT step covers "open post, use Generate Meta Description button"; verifies ability-first works |
| Contrast test fails on unrelated palette refactor | Low | Test asserts BASELINE + tolerance window; intentional brand evolution requires updating BOTH theme.json AND the test in the same commit (forces conscious decision) |
| theme.json v3 → v4 schema bump arrives during v9.5.0 cycle | Low (unknown timeline) | Audit notes it as SCHEMA-CHANGE candidate; migration deferred to v10.0.0 |
| Cross-package listener tests fail because of mock drift | Medium | Mock stubs follow the exact same minimal-WP-mock pattern as `tests/abilities-integration.php`; same maintainer surface |
