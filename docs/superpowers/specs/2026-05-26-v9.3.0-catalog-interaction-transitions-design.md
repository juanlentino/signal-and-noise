## 1. Headline

**Pillar nav + pagination transitions for /notes catalog.** v9.2.0 shipped the catalog → single-note title morph. v9.3.0 extends View Transitions to the *other* two trafficked /notes catalog interactions: filtering by pillar (e.g., "all" → "provenance"), and clicking pagination (page 1 → page 2). Both currently flash-replace; v9.3.0 makes them feel intentional.

---

## 2. Why this minor exists

Real-use feedback from v9.2.0 (assumed; revisit in kick-off brainstorm): the catalog → single-note morph feels great, but the OTHER two main interactions on /notes feel relatively dead by comparison:

- **Pillar filter clicks:** the filter buttons (likely "All / Provenance / Music / AI" or similar pillars from `inc/page-notes-render.php`) cause a full content swap with no transition. Jarring against the polished catalog → single-note morph.
- **Pagination:** /notes catalog paginated; page-to-page transitions are also full reloads.

Both are pure-CSS View Transitions extensions of the v9.0.0 + v9.2.0 infrastructure. No JS state management — relies on the existing `@view-transition { navigation: auto; }` block in `assets/css/critical.css` to enable cross-document VT, plus per-element `view-transition-name` to control which elements morph vs. fade.

## 3. Investigation needed during kick-off brainstorm

Open questions to resolve before writing the plan:

1. **Pillar nav mechanism:** are pillar filters server-side links (full page reload to `/notes?pillar=provenance`) or client-side filters (JS hides/shows DOM)?
   - If server-side: View Transitions Level 2 handles cross-document automatically once we add `view-transition-name` to the persistent elements (catalog grid wrapper, header).
   - If client-side: needs the JS-driven [`document.startViewTransition()`](https://developer.mozilla.org/en-US/docs/Web/API/Document/startViewTransition) API call wrapping the DOM mutation. ~30 LOC of JS.
   - **Action during brainstorm:** read `inc/page-notes-render.php` + `templates/page-notes.html` to determine which.

2. **Pagination mechanism:** same question — is /notes catalog paginated via server-side links (`?paged=2`) or client-side?
   - Same decision branch as pillar nav.
   - **Action during brainstorm:** check the catalog template.

3. **What persists vs. what morphs vs. what fades?**
   - **Persistent** (survives transition unchanged): page header, footer, the pillar filter row itself, "Notes" page title, the catalog wrapper.
   - **Morphs** (smoothly animates from one state to another): page-numeral indicator? Filter-active state pill?
   - **Fades** (cross-fade between states): the catalog grid contents (the actual cards).
   - **Decision needed during brainstorm:** confirm this taxonomy matches the brutalist aesthetic. Cross-fade is gentler than slide; slide is more confident but risks feeling Apple-y on a NIN-brutalist site.

4. **Mobile carveout:** does the same `@media (max-width: 768px)` disable apply to filter + pagination transitions? Filter transitions on mobile (where pillar buttons might wrap to multiple lines) could feel especially janky.

## 5. Architecture

### 5.1 Pure-CSS path (assumes server-side pillar nav + pagination)

If `/notes?pillar=provenance` and `/notes?paged=2` are real server-rendered URLs, then v9.2.0's `@view-transition { navigation: auto; }` already enables cross-document transitions for these navigations. v9.3.0 just adds:

```css
/* v9.3.0: persistent shell for catalog interactions */
.sn-notes-catalog__header  { view-transition-name: sn-notes-header; }
.sn-notes-catalog__filters { view-transition-name: sn-notes-filters; }
.sn-notes-catalog__grid    { view-transition-name: sn-notes-grid; }
```

The browser auto-figures out that the header + filter row stay put while the grid contents cross-fade. ~10 LOC CSS, zero JS.

### 5.2 Hybrid path (if filters are client-side, pagination is server-side)

If pillar filters are AJAX/JS-driven (DOM mutation, no URL change), the CSS-only approach won't fire. Need JS to wrap the filter-state mutation:

```js
function applyPillarFilter(pillar) {
  if (!document.startViewTransition) {
    // No VT support — just mutate, no animation.
    mutateGridForPillar(pillar);
    return;
  }
  document.startViewTransition(() => mutateGridForPillar(pillar));
}
```

~30 LOC JS, plus the same CSS for view-transition-name assignments. Server-side pagination still works via the cross-doc VT.

### 5.3 Reduced motion

Inherits the existing `@media (prefers-reduced-motion: reduce) { @view-transition { navigation: none; } }` block from v9.0.0. No new motion-preference code required.

For the JS path (5.2), `document.startViewTransition` itself respects reduced-motion at the browser level — when motion is reduced, the callback runs but the animation phase is skipped. No additional guard needed.

## 6. Risk register

| Risk | Severity | Mitigation |
|---|---|---|
| Pillar filter transition feels over-eager on a brutalist site | Medium | Subtle cross-fade only; no slide/scale. Test during kick-off with real /notes content; revert if it feels wrong. |
| Mobile pillar buttons wrap to multiple lines; transition looks janky during reflow | Medium | Mobile carveout via `@media (max-width: 768px)` disable, same pattern as v9.2.0 |
| Browser support: Firefox doesn't support cross-document VT yet (March 2025) | Low | Firefox falls back to instant nav (current behavior). Zero regression. |
| Pagination URL changes cause WP to reload the entire catalog template — VT might fire on the WRONG elements | Low | The persistent-element naming (`sn-notes-header`, `sn-notes-filters`) ensures only the grid contents fade. Browser handles the rest. |
| Client-side filter implementation needs JS testing infrastructure that doesn't exist yet | Medium | If hybrid path is needed, ship the JS as a small file; manual testing only (no JS test harness in this project). |
