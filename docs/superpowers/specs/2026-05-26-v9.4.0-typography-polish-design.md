# v9.4.0 ÔÇö Typography polish (justified text + hyphenation + hanging punctuation)

**Status:** SPEC LOCKED via brainstorm 2026-05-26. Three components confirmed; smart quotes intentionally dropped (wptexturize handles natively). Next: `superpowers:writing-plans` to expand into a TDD-paced implementation plan.

**Repo:** `signal-and-noise` theme (plugin untouched).

**Cap math:** theme currently at v9.3.0 (patch 0/7). This minor rolls to v9.4.0. Theme minor cap usage: 4/6 ÔåÆ **5/6** (v9.0, v9.1, v9.2, v9.3, v9.4 used; **1 minor remaining before v10.0.0**). Patch cap resets to 0/7 for v9.4.x.

**Lineage:** direct sequel to v9.3.0. The v9.3.0 narrative was *"make /notes posts feel like a printed editorial spread"* ÔÇö drop caps, footnotes, sidenotes, frontmatter spec card. v9.4.0 extends the same narrative *one layer deeper*: now that the page LAYOUT reads as editorial, the **typography of running prose** should match. Justified body text, hyphenated word-breaks, hanging punctuation at text edges.

---

## 1. Headline

**v9.4.0 is typography of running prose.** Three CSS rules applied universally to body paragraphs in single-note (`/notes/<slug>/`) posts:

1. **Justified text** (`text-align: justify`) ÔÇö body paragraphs are justified at ÔëÑ1024px viewport. Mobile falls back to left-aligned (cascade default) to avoid ragged-spacing.
2. **Hyphenation** (`hyphens: auto`) ÔÇö improves justified rivers/rag. Same responsive gating. Relies on `<html lang="en-US">` from WP's `language_attributes()` (default).
3. **Hanging punctuation** (`hanging-punctuation: first last allow-end`) ÔÇö opening/closing punctuation hangs into the margin. Safari supports; Chrome/Firefox partial-or-none ÔÇö graceful degradation (punctuation stays inside the edge, no visual breakage).

**Total scope: ~30-60 LOC of CSS in `assets/css/critical.css`, plus CHANGELOG + version bump. No JS. No PHP. No new files.**

## 2. Why this minor exists

v9.3.0 shipped the editorial-spread *layout* (drop caps, footnotes, sidenotes, frontmatter). But the body text within that layout is still WP-default: left-aligned, no hyphenation, punctuation flush to the text edge. Compared to the rest of the page ÔÇö which screams "printed editorial" ÔÇö the running prose now feels like a generic WordPress post bolted into a brutalist frame.

Typography polish closes that gap. The trio (justify + hyphenate + hang) is what distinguishes "blog post" from "magazine column." It's the most-visible-per-byte improvement available in the current design surface ÔÇö the kind of detail that gets unconsciously noticed by readers but consciously appreciated by typographers.

**Why not in v9.3.0:** v9.3.0 was about adding *new* editorial elements (footnotes, sidenotes, frontmatter). v9.4.0 is about refining the *existing* prose surface. Different abstraction layer ÔÇö cleaner narrative to ship as a separate minor.

**Why not in a patch:** rolls structural CSS that touches every existing /notes post. Behavior change on every paragraph qualifies as a minor (visible new capability), not a patch (no behavior change). Per [VERSIONING.md](../../VERSIONING.md): "MINOR for new user-visible capabilities."

## 4. Architecture

### 4.1 File location

All v9.4.0 CSS lands in [`assets/css/critical.css`](../../assets/css/critical.css) immediately after the v9.3.0 long-form post layout section (currently ending around line 819). New section header:

```css
/* ============================================
   v9.4.0 ÔÇö Typography polish (justified + hyphenation + hanging punctuation)
   ============================================ */
```

Same file as v9.3.0 because:
1. **Same scope** ÔÇö single-note posts, same `.single-post` body class.
2. **Same critical-path priority** ÔÇö typography rules affect first-paint layout; belongs in critical CSS.
3. **Easy cross-reference** ÔÇö drop caps + typography polish read as one cohesive editorial-prose system when adjacent in the file.

### 4.2 Component 1 ÔÇö Justified text

```css
@media (min-width: 1024px) {
    .single-post .wp-block-post-content > p {
        text-align: justify;
    }
}
```

**Selector reasoning:**
- `.single-post` body class ÔÇö added by WP automatically on single-post templates. Matches /notes posts.
- `.wp-block-post-content` ÔÇö the WP block-editor wrapper for post body content. Standard FSE wrapper.
- `> p` (direct-child only) ÔÇö critically excludes paragraphs nested in blocks like pull-quote (`<aside class="sn-pattern-pull-quote"> <p>`), sidenotes, footnote list items, code blocks. Those keep their own alignment.
- `@media (min-width: 1024px)` ÔÇö gates to desktop+ widths. At <1024px, the rule doesn't apply, body paragraphs cascade to left-aligned (the WP default).

**Why 1024px:** justified text needs enough horizontal space for the algorithm to balance word spacing without large gaps. Below ~50ch (roughly tablet portrait), justified text creates visibly bad rag. 1024px corresponds to the existing breakpoint pattern in `assets/css/responsive.css` for desktop styles (verify during plan phase).

### 4.3 Component 2 ÔÇö Hyphenation

```css
@media (min-width: 1024px) {
    .single-post .wp-block-post-content > p {
        text-align: justify;
        -webkit-hyphens: auto;
        hyphens: auto;
    }
}
```

Combined with Component 1 (same selector). `hyphens: auto` tells the browser to insert soft hyphens at appropriate word-break points. Browser uses the document's `lang` attribute (set by `language_attributes()` in [`parts/header.html`](../../parts/header.html)) to pick the dictionary.

**The `-webkit-hyphens` prefix is non-negotiable** ÔÇö Safari (both macOS and iOS) still requires it as of 2026. Without the prefix, Safari users see no hyphenation while Chrome/Firefox users see hyphenation, creating a visible split in how the same post renders across browsers. Always pair both declarations.

**Verification during plan phase:** confirm `<html lang="en-US">` is actually output. If not (e.g., if WP's `language_attributes` is filtered/removed), hyphenation will silently no-op. Quick check: `curl https://juanlentino.com/notes/ | grep '<html'`.

**Browser support:**
- Firefox: full support (unprefixed) since 2012.
- Chrome / Edge (Chromium-based): full support (unprefixed) since 2020.
- Safari / iOS Safari: full support, but **requires `-webkit-` prefix** as of 2026-05.
- Older browsers cascade to no hyphenation (graceful degradation).

### 4.4 Component 3 ÔÇö Hanging punctuation

```css
.single-post .wp-block-post-content {
    hanging-punctuation: first last allow-end;
}
```

**Not** wrapped in `@media`. Applies at all viewport widths because:
- The effect is enhancement-only ÔÇö opening quotes hang into the left edge of the text, closing quotes hang into the right edge, periods/commas at line-ends can hang.
- Graceful degradation is automatic ÔÇö unsupported browsers just ignore the property; no fallback rule needed.
- No layout-shift risk ÔÇö punctuation that hangs is rendered as glyph extension, not as additional text box.

**Browser support reality (as of 2026-05):**
- Safari (macOS + iOS): full support since 2017 (Safari 10). Renders correctly.
- Chrome (desktop + Android): no support as of writing. Punctuation stays inside.
- Firefox: no support. Same.
- Edge: follows Chromium; no support.

**Design implication:** Safari users see the polished version; Chrome/Firefox/Edge users see the current "punctuation flush to edge" state ÔÇö unchanged from today. **This is acceptable**: hanging punctuation is the smallest of the three rules and ships as progressive enhancement only.

### 4.5 Selector specificity defenses

Three risks where v9.4.0's rules could accidentally override v9.3.0 or v9.2.0 work:

| Risk | Defense |
|---|---|
| Drop cap's `:first-of-type::first-letter` floated left could fight justified text | `text-align: justify` doesn't break `float: left`; the dropped letter floats independent of the surrounding paragraph's alignment. Tested mentally; verify visually during smoke. |
| Pull-quote pattern (`<p class="sn-pull-quote__body">`) inherits justification | Pull-quote `p` is INSIDE `<aside class="sn-pattern-pull-quote">` (a wp:group), NOT a direct child of `.wp-block-post-content`. The `> p` direct-child selector excludes it. Verified by re-reading [theme/patterns/pull-quote.php](../../patterns/pull-quote.php) markup shape. |
| Sidenote pattern's body text inherits hyphenation | Sidenote is `<aside class="sn-sidenote">` containing its own `<p>` ÔÇö also nested, not direct child. Excluded by `> p`. |
| Steps-enumerated pattern's list items inherit justification | List items aren't `<p>` ÔÇö they're `<li>`. Excluded by `p` element selector. |
| Footnote-list items (`.wp-block-footnotes li`) inherit | Same ÔÇö not `<p>`. Excluded. |
| Frontmatter spec card (`parts/post-frontmatter.html`) inherits | Spec card is rendered OUTSIDE `.wp-block-post-content` (it lives above the post-title in `templates/single.html`). Selector doesn't reach it. |

**All risks resolved by the `> p` direct-child constraint.** During plan phase, verify by reading the actual rendered HTML of a /notes post (e.g., `curl https://juanlentino.com/notes/<post>/`).

### 4.6 Data flow

CSS-only. The browser:
1. Fetches `critical.css` (already in the existing payload).
2. Matches selectors against rendered DOM on /notes pages.
3. Applies `text-align: justify` + `hyphens: auto` to direct-child `<p>` elements within `.single-post .wp-block-post-content` (at ÔëÑ1024px).
4. Applies `hanging-punctuation` to the post-content wrapper.

Zero runtime cost. Zero new HTTP requests. Zero JavaScript.
