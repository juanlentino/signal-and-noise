# Theme v10.0.0 — Public Surface Audit + Scope

**Status:** Audit captured 2026-05-27 during theme v9.5.0 cycle. Mirrors plugin's `2026-05-26-v5.0.0-scope.md` shape.

**Purpose:** classify every public-surface item in the theme as KEEP / RENAME / REMOVE / SCHEMA-CHANGE. The classification determines what a future v10.0.0 (which only convenes on actual breaking changes per cap-drop) would have to ship.

**Expected outcome (per spec §3):** mostly KEEP. Plugin v5.0.0 scope's audit (separate doc in plugin repo) follows the same shape. Schema candidates: theme.json v3 → v4 if/when WP ships v4 schema.

**This is NOT a v10.0.0 plan.** It's an inventory that informs *whether* v10.0.0 needs to convene at all, and *what* it would carry if so.

## 11. Methodology notes

The inventory was gathered via:
- `grep -hE "^function sn_" inc/*.php | sort -u` for function discovery (36 functions)
- `grep -nE "apply_filters\(|do_action\(" inc/*.php` for dispatched hooks (4 real dispatches, 1 docblock-only match excluded)
- `grep -hE "add_(filter|action)\(\s*['\"]sn_" inc/*.php` for listened-to hooks (4 contracts)
- `grep -hE "wp_register_ability" inc/*.php` for abilities (12 registrations)
- `ls templates/ parts/ patterns/` for FSE artifacts (13 templates, 4 parts, 6 patterns)
- `head -10 theme.json | grep -i version` for schema version (v3)

Classification was done by reading each file's surface and judging whether the contract is load-bearing (KEEP), misnamed (RENAME), unused (REMOVE), or signature-changing (SCHEMA-CHANGE). For every function, callers were confirmed by grep before assigning KEEP — no "default KEEP without verification" was applied.

**Notable audit finding:** `sn_theme_restore_git_backup` has no external callers and is only used within `inc/wp-update-git-preservation.php` itself. It is technically an internal implementation detail, not public API. However, renaming it would not constitute a breaking change per SemVer (no third party can depend on it), and there is no pressing reason to rename it. Classified KEEP.

**Notable gap:** `signal-noise/sidenote` pattern is in `patterns/` but absent from `tests/patterns-registry.php`. The pattern itself is valid (confirmed via slug header). The test file should be updated in a future v9.5.x or v9.6.x patch. Not a v10.0.0 driver.

When a future audit re-runs this same methodology, the diff will surface real changes (functions added/removed, contracts evolved, etc.) rather than noise.
