# Theme v9.0.0 — design spec

**Project:** Signal & Noise theme (`juanlentino/signal-and-noise`)
**Target release:** theme **v9.0.0** (minor + patch cap rollover from v8.5.7 — additive only, no breaking changes)
**Phase:** WP 7.0 "Armstrong" alignment + browser-native modernization
**Date:** 2026-05-20
**Status:** Approved — proceed to writing-plans

---

## 1. Purpose

Bring Signal & Noise theme up to WP 7.0 "Armstrong" (shipped today, 2026-05-20) with the smallest coherent feature bundle that ships visible improvements:

1. **Dimensions opt-in** — block-level height + width support + spacing presets, giving editors layout control without writing custom CSS
2. **textIndent opt-in** — Paragraph block gains text-indent typography control
3. **CSS-only View Transitions** — browser-native page-nav motion polish (Chrome/Edge 111+, Safari 18+, ignored elsewhere; respects `prefers-reduced-motion`)

**Why these three:** Per the WP 7.0 Field Guide dev-note audit performed during brainstorming, View Transitions in WP 7.0 are **admin-only** (not frontend), and Block Visibility's theme.json integration is **deferred to WP 7.1**. The genuine theme-actionable WP 7.0 features in scope are dimensions + textIndent. The CSS-only View Transitions are browser-native (not WP 7.0) but ship the "smooth page nav" UX the user expected from the WP 7.0 marketing — the WP project itself uses VT for admin, so we adopt the same browser feature for the frontend independently.

**Out of scope (explicit deferrals):** Customizable Navigation Overlay rebuild (v9.1+), Native Breadcrumbs adoption (needs design judgement on placement), Block Visibility theme.json integration (WP 7.1+ only), Pattern Overrides (SN has no custom blocks), useRootPaddingAwareAlignments evaluation (v9.0.1 patch material), further critical.css pruning (diminishing returns post-v8.5.6).

## 2. Architecture

3 independent additions, all theme-side, all gracefully degrading:

```
theme.json (+~30 LOC)              →  WP 7.0 dimensions + textIndent opt-in
assets/css/critical.css (+~40 LOC) →  @view-transition rule + name annotations
                                       + prefers-reduced-motion guard
```

**No new PHP. No new JS. No template changes. No new blocks.** Header / footer parts stay untouched.

## 3. Component details

### 3.1 Dimensions opt-in

Adds 3 new top-level keys under `settings`:

```json
"dimensions": {
  "width": true,
  "height": true,
  "dimensionSizes": [
    { "name": "Hairline", "slug": "hairline", "size": "1px" },
    { "name": "Short",    "slug": "short",    "size": "20rem" },
    { "name": "Medium",   "slug": "medium",   "size": "32rem" },
    { "name": "Tall",     "slug": "tall",     "size": "48rem" }
  ]
}
```

**Rationale for the 4 preset sizes:**
- **Hairline (1px)** — for divider-style block widths/heights aligned with the brutalist aesthetic (sharp horizontal/vertical rules)
- **Short / Medium / Tall (20/32/48rem)** — geometric progression matching the SN typography rhythm. These slot naturally into hero blocks, sidebars, and content callouts

**Reference:** [Dimensions Support Enhancements in WordPress 7.0](https://make.wordpress.org/core/2026/03/15/dimensions-support-enhancements-in-wordpress-7-0/).

**Backwards compatibility:** None. These are additive theme.json keys. WP 6.x ignores them (the wp/7.0 schema already silences forward-compat warnings).

### 3.2 textIndent opt-in

Single boolean addition under `settings.typography`:

```json
"typography": {
  "textIndent": true
}
```

Currently only the core Paragraph block supports textIndent. Other blocks ignore the setting. Editor gains a "text indent" control in the Typography panel for paragraphs.

**Reference:** [New Block Support: Text Indent](https://make.wordpress.org/core/2026/03/15/new-block-support-text-indent-textindent/).

**Backwards compatibility:** None. Opt-in setting; defaults to no indent on existing content.

### 3.3 CSS-only View Transitions

Appended to `assets/css/critical.css`:

```css
/* ─── View Transitions (browser-native, not WP 7.0) ───────────────────
 *
 * Cross-document View Transitions are a browser CSS feature (CSS Spec:
 * https://drafts.csswg.org/css-view-transitions-2/). WP 7.0 uses them
 * for wp-admin navigation but does NOT enable them for the site
 * frontend — that's a theme-side opt-in.
 *
 * Browser support: Chrome/Edge 111+ (March 2023), Safari 18+ (Sept
 * 2024). Older browsers silently ignore the @view-transition at-rule
 * and the view-transition-name property. No JS fallback needed.
 *
 * Respects user motion preferences via the standard prefers-reduced-
 * motion media query.
 * ───────────────────────────────────────────────────────────────────── */

@view-transition {
  navigation: auto;
}

/* Persistent shell — header + footer don't re-fade on every nav. */
.sn-header { view-transition-name: sn-header; }
.sn-footer { view-transition-name: sn-footer; }

/* Pin a name on the main content area so future tuning has a hook. */
main { view-transition-name: sn-main; }

@media (prefers-reduced-motion: reduce) {
  @view-transition {
    navigation: none;
  }
}
```

**Element targets (all verified present in current theme):**
- `.sn-header` — exists in `parts/header.html` (`<div class="wp-block-group sn-header has-void-background-color has-background" ...>`)
- `.sn-footer` — exists in `parts/footer.html` (same class pattern as header — verified during brainstorm)
- `main` — emitted by WP block themes as the standard content wrapper

**Backwards compatibility:** None. CSS-only; unsupported browsers ignore the unknown at-rule. Older critical.css sites continue working.

## 5. Testing

### 5.1 Theme.json validation

```bash
# WP-CLI from a connected install (Cloudways SSH):
wp theme.json validate
```

Or visually: Site Editor → Styles → confirm no schema warnings.

### 5.2 Editor smoke test

1. wp-admin → Site Editor → open any template
2. Insert a Paragraph block → confirm a "Text indent" control appears in the Typography panel
3. Insert a Group block → confirm "Width" + "Height" controls appear in the Dimensions panel
4. Confirm the 4 dimensionSize presets (Hairline / Short / Medium / Tall) appear in the size-picker dropdown for width/height

### 5.3 Frontend View Transitions test

Modern Chrome (111+) or Safari (18+):
1. Open https://juanlentino.com
2. Click a navigation link (e.g., Home → About)
3. Confirm a smooth fade between pages (not a flash)
4. Open DevTools → Rendering → "Emulate CSS prefers-reduced-motion: reduce"
5. Reload, click nav again → confirm transition NO longer runs

### 5.4 Older browser graceful degradation

Safari 17 or older Chrome:
1. Navigate the site
2. Confirm pages load normally (no console errors, no broken layout)
3. The `@view-transition` rule is silently ignored

## 6. Explicit non-changes (preserving stability)

- **No template file changes** — all 13 templates (`templates/*.html`) and 2 parts (`parts/header.html`, `parts/footer.html`) untouched
- **No new blocks** — using existing core blocks only
- **No new PHP / `inc/` modules**
- **No new JavaScript**
- **No changes to existing dimensions or typography styles in theme.json** — only ADDS new keys
- **No removal of existing critical.css rules** — only ADDS the View Transitions block at the end
- **No changes to other CSS files** — `base.css`, `components.css`, `forms.css`, `layout.css`, `responsive.css` all untouched

## 7. Error handling / edge cases

| Failure mode | Behavior |
|---|---|
| Browser doesn't support `@view-transition` | Rule silently ignored; default page-load behavior |
| User has `prefers-reduced-motion: reduce` | Standard `@media` guard disables transitions |
| WP < 7.0 user activates the theme | `dimensions.width/height/dimensionSizes` + `typography.textIndent` settings ignored by older WP cores; theme works normally (no degradation) |
| Schema validator strict-mode | wp/7.0 schema is current; no validation errors expected |
| Editor block doesn't support textIndent | Setting ignored (only Paragraph supports it currently); editor shows no control for unsupported blocks |
