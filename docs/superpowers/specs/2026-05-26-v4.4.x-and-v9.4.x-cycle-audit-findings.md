## 3. CRITICAL — Security exposure ⚠️

### Bug-A1: `tests/contracts-smoke.php` is remote unauthenticated destructive trigger

**URL:** `https://juanlentino.com/wp-content/plugins/signal-and-noise-tools/tests/contracts-smoke.php`

**What an HTTP GET does (no auth, no nonce, no rate limit):**

1. Bootstraps WordPress via `require_once $wp_load` at [`contracts-smoke.php:19-26`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/tests/contracts-smoke.php)
2. Contract 1: `apply_filters( 'sn_purge_all_caches_result', 0, [...] )` — purges WP object cache + Breeze + Varnish (v4.4.1 correctly disabled CF + template-overrides via the args payload; this contract is now cache-purge-only)
3. **Contract 2: `apply_filters( 'sn_clear_template_overrides_result', 0 )` — Contract 2 takes NO args; the filter dispatch invokes `sn_clear_template_overrides()` in theme's `inc/template-maintenance.php:130-147`, which `wp_delete_post()`s every `wp_template` + `wp_template_part` + `wp_navigation` post in the DB.**
4. Contract 3 + 4: read-only — safe
5. Returns plain-text response leaking site URL, theme name, theme version, contract pass/fail counts

**Severity:** CRITICAL. Any internet visitor can permanently destroy Site Editor customizations + nav menu overrides + template-part overrides. No nonce, no capability, no rate limit, no authentication.

**Exposure window:** since the v4.4.0 install via wp-admin UI completed. The contracts-smoke.php file landed in production at that moment.

**v4.4.1 was supposed to fix this but missed Contract 2.** v4.4.1's CHANGELOG entry claimed the smoke runner was "safe to run repeatedly against production without unintended side effects." That claim is wrong because Contract 2 has no args payload to neutralize.

**Recommended fix (v4.4.2):**

CLI-only guard at the top of every `tests/*.php` file in both repos:

```php
if ( PHP_SAPI !== 'cli' && ! defined( 'WP_CLI' ) ) {
    http_response_code( 404 );
    exit;
}
```

Plus a `tests/.htaccess` with `Deny from all` (defense-in-depth for Apache contexts) + `deploy.yml` `tests/` exclusion (next deploy).

## 4. HIGH severity bugs

### Bug-A2: All 27 test files leak internal info via HTTP

Every `tests/*.php` file in both repos is web-accessible with no execution guard:

**Plugin (22 files in `tests/`):**
abilities-integration, admin-tabs, ai-bootstrap, audit-retention-bounds, audit-retention-setting, bot-detection, contracts-smoke, contracts-stub, cron-dashboard, cron-history, fieldset-actions-inline, health-checks, insights, legacy-url-redirect, login-intercept, pattern-adoption-apply, pattern-adoption-detect, pattern-adoption-suggest, sn-setting-update, sn-setting-update-roundtrip, theme-ability-commands, webhooks

**Theme (5 files in `tests/`):**
abilities-integration, abilities-registration, patterns-registry, post-frontmatter, view-transitions

Most of these stub WP and don't load the framework — so they're info-leak only (function names, ability slugs, capability matrices, internal data structures). `contracts-smoke.php` is the only one that boots WP (and is the destructive one in Bug-A1).

**Fix:** same CLI-only guard applied to all 27 files. Bundled with Bug-A1 fix in **v4.4.2**.

### Bug-B2: v4.3.0 pattern-adoption Suggest+Apply is non-functional in the UI

v4.3.0 added `pattern_adoption_pull_quote` + `pattern_adoption_steps_enumerated` entries to the `ABILITY_BY_CHECK` map in [`assets/health-suggest-actions.js`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/assets/health-suggest-actions.js) — but the input-building branches in `onSuggestClick` (lines 372-401) and the apply-input shape in `onApplyClick` (lines 683-689) were never updated to handle the new check types.

**Effect:**
- Click Suggest button on a pattern-adoption opportunity → `input = {}` → `callAbility('pattern-adoption-suggest', {})` fails schema validation → error rendered or silent fail
- Apply path uses drift-shaped input (`phrase`, `position`, `replacement`, `fingerprint`, `context_snippet`) instead of pattern-adoption shape (`post_id`, `block_fingerprint`, `replacement_markup`, `pattern_type`)

**Why it slipped:** 87 PHP-level assertions across 4 test files passed; zero JS-level tests existed for the dispatcher. CHANGELOG explicitly says v4.3.0 JS changes were just "2 new entries in `ABILITY_BY_CHECK` map + new `onDismissClick` handler" — input-building was missed entirely. Browser smoke never ran (live install was "owed" but not performed).

**Fix:** ~30 LOC JS — add pattern-adoption input-building branches to both handlers + a jsdom-style smoke test. Ships as **v4.4.2** OR **v4.4.3** depending on bundling preference.

## 5. MEDIUM severity bugs

### Bug-B1: Login intercept Branch 3 lacks `/wp-admin` sub-path allowlist

[`inc/login-hide.php:235`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/login-hide.php) Branch 3 fires on `strpos( $request_uri, '/wp-admin' ) === 0` for unauthenticated requests, but matches `/wp-admin/admin-ajax.php`, `/wp-admin/admin-post.php`, `/wp-admin/async-upload.php`, `/wp-admin/load-styles.php`, `/wp-admin/load-scripts.php`.

The `plugins_loaded` allowlist (lines 159-166) skips setting `$GLOBALS['sn_login_*']` flags for these paths, but Branch 3 runs independently and 404s them.

**Current impact:** NONE — no first-party plugin or theme uses `wp_ajax_nopriv_*` handlers.

**Future impact:** any plugin that ships a public AJAX endpoint (newsletter signup, contact form, lazy-load triggers) will silently 404 for logged-out visitors.

**Fix:** 5-LOC allowlist re-application at the top of Branch 3. Ships as **v4.4.3** or **v4.4.4**.

### Bug-E1 (U-09): Article schema hardcodes `inLanguage: "en-US"` ignoring locale setting

[`inc/seo-schema.php:145`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/seo-schema.php) emits `inLanguage: "en-US"` as a hardcoded string on the Article schema, but the WebSite schema (line 83) correctly reads from `sn_setting('identity.locale')`.

**Effect:** if Identity → Locale is ever changed from `en_US` to anything else, WebSite schema reports the new locale but Article schema lies about language being en-US.

**Current impact:** NONE — locale is still `en_US` per default.

**Fix:** 2 LOC — change line 145 from `'en-US'` to `str_replace('_', '-', sn_setting('identity.locale', 'en_US'))`. Ships as **v4.4.3** or **v4.4.4**.
