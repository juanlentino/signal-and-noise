# v9.2.0 — Pattern library expansion + View Transitions catalog morph

**Status:** APPROVED via brainstorming (2026-05-26). Ready for `superpowers:writing-plans`.

**Repo:** `signal-and-noise` theme (plugin stays at v4.2.1 — separate concern).

**Cap math:** theme currently at v9.1.7 (patch cap 7/7 MAX). This minor rolls to v9.2.0. Minor cap usage 2/6 → 3/6 (v9.0, v9.1, v9.2 used). Patch cap resets to 0/7 for v9.2.x. Plugin untouched.

## 1. Headline

**Three new block patterns tuned to your /notes content, one auto-rendered post-closing template part, and a catalog → single-note View Transitions morph. Additive structure only — zero changes to existing post content.**

---

## 2. Why this minor exists

Two motivating drivers:

1. **`list-block-patterns` ability is empty.** v9.1.7 (B-11 theme abilities split) registered the `signal-and-noise/list-block-patterns` ability, but with only 2 patterns existing (`hero-dossier`, `section-constrained`), the ability returns nothing useful to AI agents. v9.2.0 expands the pattern corpus to 5 total — enough for the ability to actually serve as a brand-aligned content-generation surface.

2. **The catalog → single-note flow is the most-trafficked path on the site and feels janky.** v9.0.0 already opted into cross-document View Transitions globally (`@view-transition { navigation: auto; }` in `assets/css/critical.css`), with persistent shell on header/footer/main. What's missing is the *signature element-to-element morph* — clicking a note card in /notes should cause the card title to morph into the article hero, not flash-replace. This is a high-visibility polish gain confined to one navigation path.

---

## 3. Investigation summary (already complete)

This spec's brainstorming pass established:

- **Content patterns to support** were derived from real /notes content (10 most recent post titles + body of `detection-scales-the-wrong-way`). Three recurring structural needs surfaced: (a) emphasized thesis statements, (b) X-vs-Y analytical framings, (c) enumerated multi-step reasoning ("Five layers", "Five years"). Plus one cross-cutting need: every post should have a consistent closing block with date + tags + cross-link to the next post in cluster.
- **What we deliberately did NOT include**: CTA / Subscribe (no newsletter exists), FAQ accordion (/provenance is essay format), image-with-caption (core/image already handles it), generic blockquote (core/quote already works; pull-quote is the meaningful addition), aside/sidenote (semantic-redundant with pull-quote variant if needed).
- **View Transitions scope**: minimal — catalog → single-note morph only. Pillar nav transitions + pagination transitions + cluster "next" transitions deferred (risk of feeling over-animated on a brutalist site; revisit in v9.3.0 based on real-use feedback).
- **post-closing decision**: template part (auto-renders on every post), not a manual pattern. Existing posts get the closing block retroactively but post body content is never touched — purely additive template wrap.

## 5. Architecture

### 5.1 Pattern registration

WordPress block themes auto-register patterns from any `.php` file in the theme's `patterns/` directory. No PHP registration code needed — WordPress reads the docblock header at the top of each file.

**File template** (each of the 3 new patterns follows this shape):

```php
<?php
/**
 * Title: Pull-quote
 * Slug: signal-noise/pull-quote
 * Categories: text, signal-noise
 * Keywords: thesis, quote, callout, emphasis
 * Description: Large brutalist pull-quote for highlighting thesis statements with optional attribution.
 * Block Types: core/quote
 * Viewport Width: 1200
 */
?>
<!-- wp:group {"className":"sn-pattern-pull-quote","style":{...}} -->
<div class="wp-block-group sn-pattern-pull-quote">
  <!-- wp:paragraph {...} -->
  <!-- /wp:paragraph -->
  <!-- wp:paragraph {...} -->
  <!-- /wp:paragraph -->
</div>
<!-- /wp:group -->
```

The docblock fields (`Title`, `Slug`, `Categories`, `Keywords`, `Description`, `Viewport Width`) are what `list-block-patterns` reads to surface the pattern to AI agents.

**Pattern category:** the `signal-noise` category is **already registered** by `inc/patterns.php` since v7.5.0 — no new file needed. The 3 new patterns just declare `Categories: signal-noise` in their docblock header and they cluster automatically.

### 5.2 The 3 patterns

#### 5.2.1 `pull-quote` (`patterns/pull-quote.php`)

Brutalist top + bottom black rules (3px solid `#111`), `#fafafa` bg, Georgia serif italic body at ~1.5rem, Courier mono small-caps attribution.

Block structure:
```
core/group (className: sn-pattern-pull-quote)
├── core/paragraph (className: sn-pull-quote__body)  ← the quote
└── core/paragraph (className: sn-pull-quote__attribution)  ← optional attribution
```

User can delete the attribution paragraph if not needed.

#### 5.2.2 `compare-columns` (`patterns/compare-columns.php`)

A vs B layout with vertical black divider. `#fafafa` bg, Courier mono A/B labels small-caps. Two equal-width columns separated by a 1px black centerline. Each column has a label + heading + body.

Block structure:
```
core/group (className: sn-pattern-compare-columns)
└── core/columns (verticalAlignment: top)
    ├── core/column
    │   ├── core/paragraph (className: sn-compare__label)  ← "A · Detection"
    │   ├── core/heading (level: 4, className: sn-compare__title)
    │   └── core/paragraph (className: sn-compare__body)
    ├── core/group (className: sn-compare__divider)  ← vertical rule, ::before pseudo
    └── core/column
        ├── core/paragraph (className: sn-compare__label)  ← "B · Provenance"
        ├── core/heading (level: 4, className: sn-compare__title)
        └── core/paragraph (className: sn-compare__body)
```

CSS for the divider lives in `assets/css/critical.css` under a `.sn-pattern-compare-columns` selector — adds a 1px black vertical line between columns via `::before` on the wrapper group.

#### 5.2.3 `steps-enumerated` (`patterns/steps-enumerated.php`)

Monospace zero-padded numerals (`01`, `02`, `03`, …) with baseline-aligned text. CSS grid `grid-template-columns: auto 1fr`. `#fafafa` bg, generous spacing.

Block structure:
```
core/group (className: sn-pattern-steps-enumerated)
├── core/paragraph (className: sn-steps__label)  ← optional section label
└── core/list (className: sn-steps__list, ordered: true, start: 1)
    ├── core/list-item × N  ← user adds/removes steps
```

CSS uses a `counter-reset` + `::marker` pattern to render the `01` `02` `03` numerals from the ordered list — no manual numbering required. User adds steps by inserting list items; numbers auto-update.

### 5.3 `post-closing` template part

**File:** `parts/post-closing.html` (new)

Block-theme template parts use HTML files with block markup. WordPress auto-discovers them.

Renders:
1. Horizontal black 1px line (top border)
2. Date + tags row (Courier mono small-caps)
3. "Next in cluster" cross-link block (red accent `#cc1414`, Courier mono "Next →" label, Georgia serif post title)

Block structure (inside the .html file):
```html
<!-- wp:group {"className":"sn-post-closing","tagName":"footer"} -->
<footer class="wp-block-group sn-post-closing">
  <!-- wp:post-date {"className":"sn-post-closing__date","format":"Y-m-d"} /-->
  <!-- wp:post-terms {"term":"post_tag","className":"sn-post-closing__tags","separator":" · "} /-->
  <!-- wp:group {"className":"sn-post-closing__next"} -->
  <div class="wp-block-group sn-post-closing__next">
    <!-- wp:paragraph {"className":"sn-post-closing__next-label"}-->
    <p class="sn-post-closing__next-label">Next →</p>
    <!-- /wp:paragraph -->
    <!-- wp:[CUSTOM next-in-cluster block — see below] -->
  </div>
  <!-- /wp:group -->
</footer>
<!-- /wp:group -->
```

**"Next in cluster" resolution** — there's no core block that emits "next post in same tag-cluster", so this requires either:
- (a) A dynamic block render callback registered in `inc/blocks-post-closing.php` (new file) that queries posts sharing >=1 tag with current post, sorted by date.
- (b) A simpler approach: just use core/post-navigation-link with `type: next` — gives chronologically-next post, no tag filtering. Less precise but zero new code.

**Recommendation:** ship (b) in v9.2.0 (zero new PHP), evaluate in real use whether the chronological-next is "close enough", and ship (a) as v9.2.1 if cluster-aware navigation is needed.

### 5.4 `single.html` template update

Append the post-closing template part call inside the existing single.html `<main>` block:

```html
<!-- wp:template-part {"slug":"post-closing","tagName":"footer"} /-->
```

Position: directly below the post content block, inside the same article wrapper. Zero changes to anything else in the template.

### 5.5 View Transitions: catalog → single-note morph

**Concept:** Each note card in /notes catalog gets a unique `view-transition-name` derived from the post slug. The single-note article hero gets the matching `view-transition-name`. When the user clicks a card, the browser morphs the card position/size into the article hero position/size.

**CSS** (added to `assets/css/critical.css` in the existing View Transitions block):

```css
/* v9.2.0: catalog→single-note morph. Each note card's title gets a
   unique transition-name; the single-note hero matches that name. */
.sn-note-card__title {
	view-transition-name: var(--sn-vt-card, none);
}

@media (max-width: 768px) {
	/* Disable the morph on small screens where the card→hero geometry
	   change is too dramatic and feels jarring. Fall back to the v9.0.0
	   cross-document default. */
	.sn-note-card__title { view-transition-name: none; }
}
```

**Implementation: `render_block` filter on `core/post-title`** (in `inc/blocks-view-transitions.php`, new file):

Block themes render post titles via `core/post-title` — in both the /notes catalog (inside a query loop) and the single-note article hero. A filter on `render_block` for that block type injects the inline style on every render:

```php
add_filter( 'render_block_core/post-title', function( $block_content, $block, $instance ) {
    $post_id = $instance->context['postId'] ?? get_the_ID();
    if ( ! $post_id ) {
        return $block_content;
    }
    $slug = get_post_field( 'post_name', $post_id );
    if ( ! $slug ) {
        return $block_content;
    }
    // Sanitize slug → CSS-safe identifier (alphanumeric + hyphens only).
    // view-transition-name spec requires CSS-identifier syntax — underscores
    // are technically valid but we normalize to hyphens for consistency.
    $sanitized = preg_replace( '/[^a-z0-9-]+/', '-', strtolower( $slug ) );
    $sanitized = trim( $sanitized, '-' );
    if ( ! $sanitized ) {
        return $block_content;
    }
    // Inject style="view-transition-name: sn-note-<slug>" into the
    // outer element of the rendered post-title block. Uses a simple
    // string replace on the first opening tag — adequate because
    // core/post-title's render output is well-defined (a heading or
    // link element at the top).
    $style = 'view-transition-name: sn-note-' . $sanitized . ';';
    if ( preg_match( '/<(h[1-6]|a)\b[^>]*>/i', $block_content, $m, PREG_OFFSET_CAPTURE ) ) {
        $tag = $m[0][0];
        if ( strpos( $tag, 'style=' ) !== false ) {
            $new_tag = preg_replace( '/style="([^"]*)"/i', 'style="$1; ' . $style . '"', $tag );
        } else {
            $new_tag = preg_replace( '/^<(h[1-6]|a)/i', '<$1 style="' . $style . '"', $tag );
        }
        $block_content = substr_replace( $block_content, $new_tag, $m[0][1], strlen( $tag ) );
    }
    return $block_content;
}, 10, 3 );
```

**Why this beats per-template editing:**
- Single hook covers both the catalog card AND the single-note hero (both render core/post-title).
- Works regardless of where in the site a `core/post-title` appears — the morph fires from any source page that links to a single note, not just the catalog.
- No template markup changes (zero risk to existing layout).
- The filter is idempotent and can be unhooked without touching any template.

Result: when the user clicks any link that navigates to a single note, the browser's View Transitions algorithm matches the source title's `view-transition-name` to the destination single-note hero's `view-transition-name` and morphs the position/size. No JS required.

**Note on safety:** if the source page doesn't have a matching `view-transition-name` (e.g., navigating from the homepage which doesn't render the title), the browser simply does the standard cross-doc transition from v9.0.0. There's no broken-morph failure mode — only "morph fires" or "no morph fires" outcomes.

**Reduced-motion respect:** the existing `@media (prefers-reduced-motion: reduce) { @view-transition { navigation: none; } }` in critical.css already handles disabling. Our per-element morph inherits that disable automatically (no transitions fire when navigation is `none`).

### 5.6 `list-block-patterns` ability — no code change

The ability impl at [inc/abilities-content.php](https://github.com/juanlentino/signal-and-noise/blob/main/inc/abilities-content.php) already enumerates whatever's in `WP_Block_Patterns_Registry`. Adding 3 patterns automatically grows the ability's output from 2 entries to 5. No PHP changes needed.

The only "change" is the test: `tests/theme-ability-commands.php` currently asserts 2 patterns; bump to 5 after v9.2.0.

## 6. Data flow

### 6.1 Pattern insertion (human authoring)

```
[User opens editor for new /notes post]
  → Block inserter shows "Signal & Noise" category with 5 patterns
  → User clicks "Pull-quote"
  → Pattern's block markup inserted into post_content
  → Post saved → renders with sn-pattern-pull-quote CSS class
```

### 6.2 Pattern discovery (AI agent)

```
[AI agent (via SN Tools plugin's AI client) wants to insert a callout]
  → Agent calls signal-and-noise/list-block-patterns ability
  → Ability returns 5 entries with Title + Description + Keywords
  → Agent picks "pull-quote" based on Keywords: thesis,quote,callout
  → Agent emits the pattern's block markup into the post draft
```

### 6.3 post-closing render

```
[Visitor opens /notes/some-post]
  → WordPress renders templates/single.html
  → single.html includes <!-- wp:template-part {"slug":"post-closing"} -->
  → WP looks up parts/post-closing.html
  → Renders date (via core/post-date) + tags (via core/post-terms) + next link
  → Hero black line + date row + next-read appear at bottom of article
```

### 6.4 View Transition morph

```
[WP renders any page containing core/post-title blocks]
  → render_block filter fires for each core/post-title render
  → Filter injects style="view-transition-name: sn-note-<slug>" into outer tag
  → Source page (e.g., /notes catalog): every card title carries the attr
  → Destination page (e.g., /notes/detection-scales-the-wrong-way): hero <h1> carries matching attr

[Visitor clicks a card on /notes]
  → Browser captures source state (card position/size, view-transition-name match: sn-note-detection-scales-the-wrong-way)
  → Browser navigates cross-document
  → Browser captures dest state (hero position/size, matching name)
  → Browser interpolates the morph
  → If prefers-reduced-motion: navigation: none → standard instant cross-doc nav (no morph)
```

## 9. Risk register

| Risk | Severity | Mitigation |
|---|---|---|
| `view-transition-name` collisions across posts | Low | Slug-derived names are unique by definition (WP enforces slug uniqueness per post type) |
| Slug contains underscores → invalid CSS identifier for `view-transition-name` | Low | Sanitize slug → replace `_` with `-` in the template helper (single regex) |
| Existing `hero-dossier` / `section-constrained` patterns conflict with the new `signal-noise` category | Low | Add them to the new category too (their existing categories stay); de-dup happens at WP's registry level |
| `post-closing` template part overlaps visually with existing content (some posts already have a manual signature) | Medium | Audit existing posts (10 most recent) before tagging; if any have manual closing blocks, decide case-by-case whether to remove them. None observed in the spec-investigation pass. |
| Browser support gap: Firefox doesn't support View Transitions yet (Mar 2025: behind flag) | Acceptable | Firefox users get standard instant cross-doc nav (current behavior). Zero regression. |
| Chrome/Safari morph looks janky for cards far from the hero target geometry | Medium | The 768px mobile breakpoint disables the morph on small screens where geometry change is most dramatic. Desktop large-screen morphs have a smoother path. |
| Pattern markup using core/list with `counter-reset` doesn't render numerals correctly on all themes | Low | This IS the theme — we control the CSS. Smoke test via manual verification (step 2 of §7.3). |
