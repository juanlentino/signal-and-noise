## Three primary additions

### 1. Pattern-adoption detector + scan flow (`inc/pattern-adoption-detect.php`, 187 LOC)

Walks `parse_blocks()` output for all published `post_type=post`. Matches:
- `core/quote` → pull-quote candidate
- `core/list` with `attrs.ordered === true` → steps-enumerated candidate

Each candidate gets fingerprint = `md5(serialize_block($node))`. Filters out fingerprints in `_snt_pattern_adoption_dismissed` post meta (array of `"<pattern_type>:<fingerprint>"` strings). Scan result cached in user-scoped transient `snt_pattern_adoption_candidates_<userid>` with 1h TTL.

REST endpoint `POST /signal-noise/v1/health/pattern-adoption-scan` triggers the scan; wired through `sn_handle_admin_post()`'s `pattern_adoption_scan` action (mirrors `health_scan` convention).

### 2. Suggest + apply pair with block-tree round-trip safety

**Suggest** (`inc/pattern-adoption-suggest.php`, 245 LOC after correction): pure deterministic template substitution. Given `(post_id, fingerprint, pattern_type)`, locates the candidate block, builds replacement markup mirroring `signal-and-noise/patterns/pull-quote.php` or `patterns/steps-enumerated.php`. Inline formatting (`<strong>`, `<em>`, `<a>`, `<code>`) preserved via `wp_kses` allowlist. NO AI calls.

**Apply** (`inc/pattern-adoption-apply.php`, 188 LOC): the load-bearing safety code. Path:
```
parse_blocks($post_content) → walk tree → find block where md5(serialize_block($node)) === fingerprint
  → mutate node in place → serialize_blocks($modified_tree) → wp_update_post
```

Operates entirely on block-tree node references — **no byte offsets, no raw-vs-stripped coordinate fragility** (sidesteps the v4.1.1 drift-phrase bug class). Code-quality reviewer ran 7 adversarial probes on the walker; all passed.

### 3. Health-tab Opportunities sub-section + Abilities API

**Opportunities renderer** (`inc/pattern-adoption-admin.php`, 165 LOC): sibling fieldset to the Problems section in the Health tab. Collapsed-by-default via HTML5 `<details>` / `<summary>`. Per-candidate review queue: post title + permalink + pattern_type pill + Suggest + Dismiss buttons. Reuses `assets/health-suggest-actions.js` via dispatch table entries (Task 6).

**Abilities API** (`inc/abilities-ai-pattern-adoption.php`, 151 LOC): 2 abilities registered — `signal-noise/pattern-adoption-suggest` (idempotent) and `signal-noise/pattern-adoption-apply` (destructive). Category `ai-generation` (consumer-focused — desktop-mode Copilot et al. look here for actionable abilities). Permission callback `snt_ability_perm_edit_post` (string callable, not closure). Per-feature file follows the v4.1.3 split convention.

## Discipline lessons codified this cycle

This session reinforced and extended existing patterns:

1. **The two-stage review discipline paid for itself twice in this minor.** Task 2's invalid block markup and Task 4's unreachable form-post hook were both caught at the code-quality review stage and corrected via follow-up commits. Without the review gate, v4.3.0 would have shipped with: (a) every applied pattern marked as "unexpected or invalid content" in the editor, OR (b) a "Scan for opportunities" button that silently no-ops. The reviews cost ~30 min total across 6 dispatches; the bugs would have cost an emergency v4.3.1 cycle + user-facing regression.

2. **Four plan-vs-reality drifts caught, all same root cause.** Plan written from external assumptions instead of reading the actual project code:
   - Task 2: assumed `signal-noise/pull-quote` was a registered block type → reality: pattern slug, verifiable by reading `patterns/pull-quote.php`
   - Task 4: assumed `add_action('admin_post_$slug', ...)` works for SN admin forms → reality: project uses custom `sn_handle_admin_post()` dispatcher, verifiable by reading `inc/admin-page.php`
   - Task 5: assumed `$GLOBALS['__test_registered_abilities']` had wrapped shape `{slug, args}` → reality: flat keying with slug as array key, verifiable by reading the test file
   - Task 6: assumed dismiss handler would be a sibling `document.addEventListener` → reality: project uses single dispatcher with `if/else if` branches inside `init()`, verifiable by reading the JS file's bottom

   **Generalization for future plan-writing:** for any task that wires into an existing project subsystem (admin pages, REST routing, dispatchers, registries), the plan should explicitly instruct the implementer to read the existing wiring before writing new code. Memory entry [feedback_read_peer_implementation_first.md] was updated this session to reinforce.

3. **Inline execution for small mechanical changes is sometimes the right call.** Task 6 (JS dispatch + dismiss handler, 41 LOC, no automated tests in project) was completed inline after the initial subagent dispatch got interrupted — faster, lower context cost, equivalent quality. The subagent-driven discipline is the default but should yield to inline when (a) change is small + mechanical, (b) files are already in context, (c) no automated test harness for reviewers to lean on.

4. **Plan LOC estimates are systematically low.** Plan estimated ~640 LOC; actual was ~1,300 LOC of plugin code + tests (2x). Drivers: WordPress-standard docblock verbosity, defensive code (`function_exists` guards, sanitize callbacks), `inc/abilities-ai-pattern-adoption.php` (151 LOC vs ~80 estimate), and 2 follow-up fix commits. Future plans should pad LOC estimates by 1.5-2x for WP projects.

5. **Plan assertion-count estimates are similarly low.** Plan estimated 55 new assertions; actual was 79. Mostly due to per-test-scenario assertion-count undercount (each scenario in the plan typically had 1-2 more assertions than the plan's "Test N: K assertions" comment claimed). Future plans should count assertions in the final test code, not in the prose.
