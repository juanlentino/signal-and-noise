## Session-level lessons captured

Three discipline lessons surfaced in this arc. All worth preserving:

### 1. Read source for COUNTS as well as API behavior

The spec initially claimed "16 plugin abilities + 15 commands" — both wrong. Actual: 17 and 17. The user caught it (*"We have 17 AI Abilities given by the plugin for example."*) Same failure mode the memory file `feedback_read_framework_source.md` warns against, just applied to numbers instead of functions. **30 seconds of `grep -c` would have prevented the propagation through spec + 2 plans + a failing test assertion.**

### 2. Question architectural inertia

The spec's first draft used plugin-proxy architecture "to match the existing v8.4.0 cross-package filter pattern." The user pushed back: *"If we added AI Abilities to the plugin that are for the theme, let's re-evaluate them."* The pattern-match was wrong — the existing 3 filters are for cross-cutting operational concerns; theme-domain reads aren't cross-cutting. The pivot eliminated 10 hypothetical filters and a 503 failure mode. **Architectural inertia is a real failure mode; the cheapest pushback time is at spec-review.**

### 3. Batched dispatch is right when work is structurally uniform

The skill's "fresh subagent per task" default would have meant 18 implementer dispatches + 36 reviewer dispatches = 54 subagent invocations. Batching reduced to 5 implementer dispatches + 2 (later 0) review dispatches. **The skill defaults to safety; pragmatic batching is correct when context is bounded AND tasks share a verified pattern.** The 12 abilities were all "register schema + execute_callback + tests" — one implementer doing them sequentially is identical in output to 12 separate implementers.
