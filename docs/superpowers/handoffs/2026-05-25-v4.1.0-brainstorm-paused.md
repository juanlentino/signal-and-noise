## Remaining brainstorm questions (for next session to ask)

Per the brainstorming skill's "one question at a time" rule, ask these sequentially:

### Q1: Combined v4.1.0 or split into v4.1.0 + v4.1.1?

- **Combined**: both stale-posts + orphaned-media ship as one v4.1.0 (~300-400 LOC). Bigger verification surface but one coherent "completion" narrative.
- **Split**: v4.1.0 = stale-posts only (~150-200 LOC, faster ship); v4.1.1 = orphaned-media (~150-200 LOC). Smaller per-ship risk; closer to project's "one feature per ship" pattern. v4.0.x cap counter goes 0/7 → 1/7 instead of 0/7 → directly to v4.1.0 minor.

**Recommended**: Combined. Both features share the Suggest+Apply pattern infrastructure; splitting adds ceremony without proportional safety value. v4.0.0 shipped 2 checks combined (alt + drift) successfully — same model.

### Q2: Stale-posts Apply scope

- **Read-only list (matches the `apply: null` sentinel from v4.0.2)**: AI generates a suggestion document listing specific outdated statements + recommended edits. User reads, opens editor, applies manually. No Apply button.
- **Partial-Apply (some safe rewrites)**: AI suggests + applies safe edits like "Updated: 2026-05" notes prepended, or adding `<aside>This post is from 2024; some references may be outdated</aside>` blocks. More complex; needs careful boundary on "safe" rewrites.
- **Full read-only with Copy** (extends v4.0.2 pattern): like v4.0.2's inline-img, but the Copy gives a list of suggested edits to paste into the editor.

**Recommended**: Read-only list (option 1). Most conservative; reuses the `apply: null` JS sentinel. Adds least new code. Partial-Apply (option 2) is high-risk for a v4.1.0 — defer if it ever ships.

### Q3: Orphaned-media confidence threshold

The v4.0.0 Health check currently flags orphaned media via: not-featured-image AND basename-not-in-any-post_content AND older-than-7-days. After AI analyzes context, the AI assigns a confidence score:

- **High confidence orphaned (>0.8)**: one-click delete button enabled
- **Medium confidence (0.5-0.8)**: shows "may be orphaned" + manual review prompt; no delete button
- **Low confidence (<0.5)**: not flagged at all (filtered out by the AI pass)

Or simpler binary: AI returns "delete" / "keep" / "unsure" → show delete for "delete", hide for "keep", show review for "unsure".

**Recommended**: Binary verdict (matches the drift_time_phrases pattern from v4.0.0). Three-state UX with confidence buckets adds complexity that real-user testing would inform; defer the confidence-bucket UX to a future iteration if needed.

### Q4: Drift cache pattern reuse?

Both new features run AI calls per post (stale-posts) or per attachment (orphaned-media). Should v4.1.0 add similar caching (like v4.0.1 did for drift)?

- **Yes, cache up-front**: add caching per `(subject_id, subject_modified, md5(prompt))` from v4.1.0 ship.
- **No, monitor first**: ship v4.1.0 without caching; watch production AI Request Logs after deploy; add caching as v4.1.x patch if redundancy surfaces.

**Recommended**: Yes, cache up-front. The v4.0.1 lesson (8-10 redundant calls per scan) generalizes. ~30 LOC per check; trivial to add now.

### Q5: User confirmation level for orphaned-media Apply (delete)

Deleting an attachment is destructive. Options:

- **Modal confirmation** (reuse v4.0.3 pattern): shows attachment thumbnail + filename + "this will delete the attachment permanently" → Cancel/Delete buttons.
- **Two-step button**: first click = "Delete?" confirmation inline, second click = actual delete.
- **Native browser `window.confirm`** (would regress UX from v4.0.3).

**Recommended**: Modal (option 1) — v4.0.3 established the modal pattern; orphaned-media Apply is exactly the use case it was built for. Reuse pays off.
