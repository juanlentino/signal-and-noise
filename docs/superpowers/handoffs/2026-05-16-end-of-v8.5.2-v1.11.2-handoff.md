## How the .git preservation works

New file in each repo: `inc/wp-update-git-preservation.php` (~200 LOC, mostly docblock). Three layers:

1. **`upgrader_pre_install`** (priority 10, accept_args=2) — atomic `rename()` `.git/` → `wp-content/upgrade/sn-<slug>-git-backup/` BEFORE WP's `clear_destination()` runs. Returns `WP_Error` to abort install if backup fails (better than silent .git destruction).

2. **`upgrader_post_install`** (priority 10, accept_args=3) — atomic `rename()` backup back into the destination dir (uses `$result['destination']`). On WP-side install failure (response is `WP_Error`), restores to the original dir so the rolled-back code's checkout stays intact.

3. **`admin_init` self-recovery** — every admin pageview, if an orphaned backup is detected (post_install never fired — PHP timeout, fatal in another upgrade hook), restore intelligently. Idempotent: next pageview after successful restore is a no-op.

**Critical primitive:** same-filesystem `rename()`. Atomic at the kernel level — no window where `.git` exists in both places or neither. Cross-FS rename silently falls back to copy+delete (NOT atomic), which is why the backup lives under `wp-content/upgrade/` (same mount as `wp-content/themes/` and `wp-content/plugins/` in standard WP installs incl. Cloudways).

**Failure mode handling:** `error_log()` for restoration failures, NOT `WP_Error`. The WP install itself succeeded; a failed `.git` restore is post-hoc and shouldn't fail the install. Manual recovery (if both post_install + admin_init fail) is a one-liner: `mv wp-content/upgrade/sn-*-git-backup wp-content/<themes|plugins>/<slug>/.git`.

## What to actually test on next update (the proof)

The filter pair is dormant on this install (it activates only when WP_Upgrader runs, and these v8.5.2 + v1.11.2 releases shipped via `gh workflow run` which is git-pull / SSH-checkout — not WP_Upgrader). Real test happens on the NEXT update:

1. Make a trivial change (CHANGELOG entry, docblock tweak — anything that bumps version).
2. Tag + push as v8.5.3 / v1.11.3.
3. **Don't run `gh workflow run`.** Wait up to 1h (plugin cache) or 12h (theme cache, until theme matches plugin's TTL drop) for WP to see the tag. Or visit `wp-admin/update-core.php?force-check=1` — works on plugin since v1.11.1.
4. Click "Update Now" in wp-admin.
5. After install completes, run `gh workflow run deploy.yml --ref v8.5.3 --repo juanlentino/<repo>` against the SAME tag. **If it succeeds, the .git preservation worked** — the destination still has `.git`, and the git-pull / git-checkout found something to update.

If step 5 fails (no .git), the preservation didn't work, and you have an orphan in `wp-content/upgrade/sn-*-git-backup/` to restore manually. The admin_init recovery on the next pageview should also try to restore it.
