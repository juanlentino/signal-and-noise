## What's still rough (and queued for v1.9.2 → v1.9.4)

A cross-tab design critique surfaced this finding: **3/8 tabs still look like raw WP form-table with inline styles, while their neighbors look polished.** Specifically:

- **Cloudflare** ([`inc/cloudflare-purge.php`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/cloudflare-purge.php), 336 LOC, 21 inline-style strings) — form-table for token + zone config; inline-styled cards for status. No "module configured / missing token" status indicator like Login has.
- **Plausible** ([`inc/plausible-admin.php`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/plausible-admin.php), 170 LOC, 23 inline-style strings) — Stats API token form (mirrors Login slug field's situation but without the constant-override-locking treatment); separate "Run Test" + "Open dashboard" cards using inline-styled flexbox.
- **Reading Time** ([`inc/reading-time.php`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/reading-time.php), 399 LOC, 14 inline-style strings) — scan + apply tool with a `widefat striped` results table. Unique pattern (it's a tool, not a settings page), needs the most care.

Navigating from Identity (polished) → Plausible (raw) feels jarring. Same problem v1.8.1 fixed for Dashboard and v1.9.0 fixed for Identity/Login.

## Priority ladder for v1.9.2 → v1.9.4

### v1.9.2 — Plausible tab redesign (recommended first)
**Why first:** smallest module (170 LOC), closest pattern-match to Login (status display + config form + action buttons), lowest risk.

**Pattern to apply:**
1. Replace the intro `echo '<p style="…">…</p>'` with `<p class="sn-prose">`
2. Add a `.sn-status-box` at the top reflecting Stats API token state — *Configured* (green) when `SN_PLAUSIBLE_STATS_TOKEN` constant is set OR `sn_pl_token` option has a value; *Missing token* (amber) otherwise. Mirror the pattern in [`inc/admin-page.php`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/admin-page.php) Login tab block (search `Module active`, `Module bypassed`).
3. Wrap the Stats API token form in `.sn-fieldset` with `.sn-fieldset-h` + `.sn-fieldset-intro`
4. Token input → `.sn-field .sn-field-w-lg` (it's a long opaque string); use `.sn-fieldset-actions` for the Save button
5. "Run Test" + "Open dashboard" cards → `.sn-card` + `.sn-card-grid` (same pattern as Dashboard action cards)
6. Test result display (currently inline-styled) → `.sn-status-box --ok` or `--err` depending on outcome
7. Preserve all existing `sn_action` POST handlers — they already route through the v1.9.0 `sn_handle_admin_post()` PRG handler

**Verification:** the Plausible tab's actions hit external APIs. After deploy, test:
- Hit `wp-admin/admin.php?page=sn-plausible` (logged in) — page renders without PHP errors
- Click "Run Test" — actual Stats API call fires, success or failure surfaces in the new status box
- Save a fake API key (e.g. `test123`) → Save → URL changes to `?sn_flash=…` (PRG works) → status box updates

### v1.9.3 — Cloudflare tab redesign
**Why second:** medium module (336 LOC); similar pattern to Plausible but with two config fields (token + zone ID) and a manual purge action button.

**Pattern to apply:** same as v1.9.2. Status box reflects "Token + zone configured" vs "Missing token" vs "Missing zone ID". Token field → `.sn-field-w-lg`; zone ID field → `.sn-field-w-md`. Manual purge button → `.sn-card` with action.

**Special considerations:** Cloudflare module has a "last purge timestamp" display — should become a `.sn-helper` text below the status box. The "purge specific zone" form vs "purge everything" forms — keep separate or merge under one fieldset? Recommend keeping separate (two `.sn-fieldset` cards) since they have different blast radii.

**Verification:** test against your actual Cloudflare zone. Manual purge should still fire correctly post-redesign.

### v1.9.4 — Reading Time tab redesign
**Why last:** largest module (399 LOC), unique pattern (scan + apply tool, not a settings page). Highest risk.

**Pattern to apply:**
1. Intro paragraph → `.sn-prose`
2. The 2-card "Scan" / "Apply" pattern is good — convert inline-styled cards to `.sn-card` + `.sn-card-grid`
3. Scan-results table currently uses `widefat striped` (WP default). Decision: KEEP it as-is. The widefat class is the right pattern for displaying a list of posts with multiple columns — overriding it with custom styling would look out of place. Just remove the inline `style="border:0;"`.
4. Result summary ("X posts found" or "✓ No legacy strings") → `.sn-status-box --ok` or default

**Special considerations:** the "Apply to X post(s)" button modifies post content destructively. Make sure the destructive-action warning copy is prominent in the redesign — possibly a `.sn-callout` with a red accent (might need to add `.sn-callout--danger` modifier).

**Verification:** test the scan first (read-only, safe). Don't test apply on real posts unless you have a backup.

## Architecture invariants to preserve in v1.9.2-v1.9.4

1. **All form POSTs route through `sn_handle_admin_post()`** in [`inc/admin-page.php`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/admin-page.php) — the central `admin_init` handler that does PRG (gotcha #19). Don't reintroduce inline `$_POST` handling in the module files. New actions get added to the `if/elseif` chain in `sn_handle_admin_post()`.

2. **Forms post to the current admin URL with `sn_action` hidden input** — the handler reads `$_POST['sn_action']`, processes, redirects with `?sn_flash=…`. The module's render code reads `$_GET['sn_flash']` and pushes notices. Existing convention — preserve it.

3. **Module files (`cloudflare-purge.php` etc.) shouldn't access `$notices` directly.** That's owned by `sn_theme_options_page()` in admin-page.php. Module render hooks fire inside the tab render block, after notices are already emitted.

4. **CSS additions go in `assets/admin.css`.** Use CSS vars (`--sn-surface`, `--sn-border`, `--sn-text-muted`, etc.). Don't reintroduce inline styles even for "just this one thing".

5. **Each tab redesign ships as its own PATCH bump** (v1.9.2, v1.9.3, v1.9.4) with a focused commit + CHANGELOG entry. Don't batch them — atomic rollback is the point.
