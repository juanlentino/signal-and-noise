Remaining items genuinely tied to WP 7.0 adaptation:
- Verify the dormant AI surface actually wakes up
- Migrate breadcrumb emission to native block (planned in roadmap)
- Register SN actions in the native Command Palette
- Phase 16 first AI feature (AI OG card title)

**Critical mid-session catch:** before deleting `sn_schema_breadcrumb_list()`, I fetched the actual Gutenberg source for `core/breadcrumbs` and discovered the native block emits **visual HTML only — no BreadcrumbList JSON-LD.** Both our `seo-schema.php` docblocks AND the absorption roadmap had inherited the same false claim that the native block would replace our structured-data emission. Deleting our code would have silently disappeared breadcrumb rich results from SERPs. Memory rule [[feedback_read_framework_source]] paid off — corrected the comments + the roadmap doc + logged it as WP-REFERENCE gotcha #30 so future-us doesn't re-believe the false claim.

Pivoted: dropped the breadcrumb migration, scoped Command Palette + AI OG card title as the only remaining "adapt" work. Consolidated into a single v2.3.0 minor release rather than two atomic releases (one less WP-UI install for the user).

## What v2.3.0 ships

### Command Palette integration (5 commands)

`@wordpress/commands` is JS-only — no PHP wrapper. Verified against [gutenberg/packages/commands/src/store/index.js](https://github.com/WordPress/gutenberg/blob/trunk/packages/commands/src/store/index.js) (store name `'core/commands'`) and [.../components/command-menu.js](https://github.com/WordPress/gutenberg/blob/trunk/packages/commands/src/components/command-menu.js) (callback signature `command.callback({ close })`).

Files: [inc/command-palette.php](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/command-palette.php) (enqueue) + [assets/command-palette.js](https://github.com/juanlentino/signal-and-noise-tools/blob/main/assets/command-palette.js) (registration).

Commands (admin-only, gated on `manage_options`):
- SN: Purge all caches → `POST /signal-noise/v1/cmd/purge-caches`
- SN: Clear template overrides → `POST /signal-noise/v1/cmd/clear-overrides`
- SN: Force-check updates → `POST /signal-noise/v1/cmd/force-check`
- SN: Show deploy status → `GET /signal-noise/v1/cmd/status` (snackbar)
- SN: Open Signal & Noise settings → navigate to dashboard

All commands hit existing REST endpoints from `inc/desktop-mode-integration.php` — one endpoint set, three callers (admin UI, desktop-mode palette, WP-native palette). Snackbars via `core/notices`. Dashicons embedded as React elements via `wp.element.createElement` (no JSX build step).

### AI OG card title generator

New filter `sn_og_card_title` at [og-card-generator.php:181](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/og-card-generator.php#L181) — default = `$post->post_title` so pre-existing posts are unchanged.

New post meta `_sn_og_card_title` (REST-exposed) with a textarea field in the per-post SN meta box. AI button generates a 60-90 char punchier variant, writes to the meta, AND immediately re-runs `sn_generate_og_card()` so the PNG on disk reflects the new title without requiring a post save.

The HTML `og:title` meta tag is **untouched** — search engines and social-share scrapers still see the canonical article title. Only the visual title baked into the card image changes.

Files:
- [inc/ai-og-card-title.php](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/ai-og-card-title.php) — REST endpoint + filter callback + script enqueue (~150 LOC)
- [assets/ai-og-card-title.js](https://github.com/juanlentino/signal-and-noise-tools/blob/main/assets/ai-og-card-title.js) — DOM-built button + status row (~120 LOC, mirrors ai-meta-description.js pattern)
- [inc/post-settings.php](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/post-settings.php) — new field + save handler + typed accessor

Provider-agnostic: no temperature/top_p/top_k anywhere. All constraints in `using_system_instruction()` — works against Anthropic + Gemini connectors without code branching.

## Workflow patterns that worked this session

1. **Source-check before deletion.** The `feedback_read_framework_source` memory rule caught the breadcrumb-block false claim before I shipped a silent SERP regression. Single WebFetch to a raw GitHub file was 30 seconds well spent.

2. **Consolidate atomic releases when scope is tight.** Original plan was v2.3.0 (adapt) + v2.4.0 (AI OG title) as two releases. Mechanically they were tested together; shipping as one cut WP-UI install passes in half. Decision was reversible — could have split if either piece had failed lint.

3. **`★ Insight` callouts forced me to articulate non-obvious design choices.** "AI runs at button click, not at card render time" is the kind of decision that would normally be implicit; writing the insight made the trade-off explicit + searchable in the transcript.

4. **Stale memory caught early.** First few tool calls revealed plugin was at v2.2.0, not v2.0.3 as memory claimed. Updated immediately rather than reasoning on bad data downstream.
