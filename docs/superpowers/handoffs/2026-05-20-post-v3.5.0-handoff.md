## Plugin releases in chronological order

### v3.0.2 — Tab registry single source of truth + a11y + i18n polish

- `sn_admin_pages()` is now the only place a tab is registered. `sn_admin_page_valid_tabs()` + `sn_admin_page_tab_labels()` derive from it via `array_column`.
- A11y: labeled filter input, table `<caption>`, scoped `<th>` cells, screen-reader-text on SN/orphan badges, `aria-label` on Run-now buttons.
- i18n: `sntCronI18n` localize bag with defensive English fallback.
- New `tests/admin-tabs.php` — 135 → 163 assertions by session end. **The Cron tab regression guard would have caught v3.0.0's 4-coordinated-edit bug before merge.**
- See `inc/admin-page.php:83-110` for the new helpers; `tests/admin-tabs.php` for the regression coverage.

### v3.1.0 — Cron Dashboard Unschedule

- New `snt_cron_unschedule_event_impl($hook, $args)` using `wp_clear_scheduled_hook()` (NOT `wp_unschedule_event()` — rationale documented in `inc/cron-dashboard.php` and the CHANGELOG).
- Per-row granularity; SN-owned hooks refused at the impl-layer guard with clear remediation text.
- 4-surface dispatch: dashboard button (with `confirm()` + row fade-out), REST `POST /signal-noise/v1/cron/unschedule`, `signal-noise/unschedule-cron-event` ability (`destructive: true`, `idempotent: true`, `open_world_hint: false`), desktop-mode unchanged.
- Tests: 35 → 54 assertions on cron-dashboard. **Test 19 (args mismatch isolation)** is the critical lock against ever swapping to args-blind clearing.

### v3.2.0 — Cron history log

- New `wp_snt_cron_history` table via `dbDelta`; rolling 30-day window + 1000-row per-hook hard cap; daily `snt_cron_history_prune` cron.
- Capture: `-PHP_INT_MAX` + `+PHP_INT_MAX` action pair registered for every hook in `_get_cron_array()` during DOING_CRON. Run-now writes directly with its own success/error from the try-catch (gated by `__snt_cron_history_skip_auto` to dedupe with the post-callback).
- Surfaces: `GET /signal-noise/v1/cron/history?hook=X&limit=N`, `signal-noise/get-cron-history` ability (readonly, idempotent), inline expand-history panel on the Cron tab's "Last fired" cell (uses `aria-expanded` for screen-reader announce).
- New `tests/cron-history.php` — 24 assertions including a self-contained wpdb stub.

### v3.4.0 — Personal automation webhooks

- New `inc/webhooks.php` (~270 LOC) — CRUD over `sn_webhooks` option store, HMAC-SHA256 signing, async dispatch via `wp_schedule_single_event`, retry on 5xx / network error (3 attempts, 5min backoff), per-webhook 20-entry rolling delivery log in `sn_webhook_log_<id>` (autoload=false).
- Trigger: `transition_post_status` with the canonical `(publish && !was_publish)` guard. Allowed types: `post`, `page` (filterable via `sn_webhook_post_types`).
- Sign: `X-SN-Signature: sha256=<hex>` over the raw body. Receivers must verify.
- Secret: 48 chars from `wp_generate_password`, shown once on create OR rotation; rest of the time obscured in the admin UI as `xxxx…xxxx`.
- New 9th admin tab Webhooks with full CRUD UI + delivery log details disclosure + payload reference panel.
- New `tests/webhooks.php` — 46 assertions across 10 tests.

### v3.5.0 — Content Health detection scans (no AI in v1)

User picked the most comprehensive scope: all 4 checks, manual-only schedule, detection-only AI flow.

- **4 checks:**
  1. Missing alt text (attachment + inline `<img>` scan; respects `alt=""` decorative-image spec)
  2. Orphaned media (>7 days old, not featured-image, not referenced by basename in post body)
  3. Broken internal links (HEAD probe with 24h per-URL transient cache, HEAD-405 → GET fallback)
  4. Stale posts (published > 12 months ago, no edits since)
- **Manual-only:** "Run scan" button. No background cron, no auto-fix, zero AI token spend.
- **24h scan-level transient cache** (`sn_health_last_scan`) so re-visiting the Health tab doesn't re-scan.
- **Forward-compatible finding shape**: future v3.5.x AI-propose work just adds a `suggestion` field per finding without breaking the admin UI consumer.
- New 10th admin tab Health.
- New `tests/health-checks.php` — 32 assertions covering the pure-logic helpers (regex extractors, link-status caching, HEAD-405 fallback, error envelopes).

## Process discipline outcomes

The hard rule ([`feedback_skills_plugins_docs_always`](../../../.claude/projects/-Users-juanlentino-Projects-signal-and-noise/memory/feedback_skills_plugins_docs_always.md)) held through all 5 releases. Every code change went through verify-before-completion. The 5-suite test grid acts as a continuous safety net: a change to `sn_admin_pages()` automatically gets tested via admin-tabs; a change to `snt_cron_unschedule_event_impl` automatically gets the args-mismatch isolation test; the regex extractors in health-checks all have spec-tracking tests.

The "Only prompt forks" directive worked cleanly. The single fork (v3.5.0 scope) was a genuine 3-axis decision (which checks × schedule × AI flow) with multiple defensible answers, where my recommendation would have been more conservative than what the user ultimately picked. Everything else had a clear-better answer that didn't need clearance.

## Lessons captured this session

| Lesson | Encoded where |
|---|---|
| v3.0.0's "4-coordinated-edit miss" is now architecturally impossible | `sn_admin_pages()` is SSOT + 163-assertion regression guard in `tests/admin-tabs.php` |
| `wp_clear_scheduled_hook` vs `wp_unschedule_event` — pick the former for "user wants to STOP this schedule" semantics | Test 18+19 in `tests/cron-dashboard.php` would catch a future swap |
| `set -u` is incompatible with optional positional args in bash heredocs | Parameter-expansion default (`${2:-}`) pattern documented in `cron-orphan-cleanup.yml` |
| WP-Admin app passwords scope per-repo | `deploy.yml` comment + remediation message + this handoff |
| Custom table is the right primitive when (a) write contention matters OR (b) retention windowing is needed | v3.2.0 cron-history + v1.x rss-plausible-tracker both follow this pattern |
| HMAC-SHA256 signing pattern matches GitHub/Stripe — receivers don't need new mental model | v3.4.0 webhooks; documented in admin UI payload reference |
| Detection-only is the right v1 scope for any health-check-like feature — adds value, defers AI cost, keeps shape forward-compatible | v3.5.0; `findings` shape ready for `suggestion` field in v3.5.x |
