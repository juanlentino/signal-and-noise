5. **`ed7c27b`** — Task D: version bump 3.6.1 → 3.7.0 + CHANGELOG + ship.

**Algorithm:** Two-stage hybrid — SQL `REGEXP` + PHP `preg_match_all` pre-filter (zero token cost) eliminates posts with no candidate time-relative phrases; AI second pass evaluates surviving candidates in context vs. the post's `last_modified` date. Only `stale` verdicts surface as Health-tab findings. Candidate count capped at 25/post to fit `max_tokens=600` (prevents silent post-drops from mid-JSON truncation). Per-scan cost: ~$0.01–$0.02. Annual ceiling at weekly use: under $1.

**Detection only in v1.** Findings deep-link to editor; AI-suggested replacements deferred to v3.7.x+.

**Net-new vs. ecosystem:** `ai/ai` v1.0.0 ships Editorial Notes (grammar / SEO / readability / a11y) but no semantic drift detection. v3.5.0's Health tab is rule-based — counts missing alts, finds orphaned media, but can't tell whether "recently" is still accurate. v3.7.0 fills that exact gap.

### The 401 — a wrinkle worth understanding

The CF purge step calls our own `/wp-json/signal-noise/v1/purge-cache` endpoint with HTTP Basic Auth using the GHA secret `WP_DEPLOY_APP_PASSWORD`. Response:

```json
{
  "code": "sn_rest_forbidden",
  "message": "You do not have permission to perform this action.",
  "data": {"status": 401}
}
```

That's `sn_rest_forbidden`, **NOT** a generic WP "incorrect username or password" 401. Meaning:

1. Basic Auth got accepted at the WP REST level (otherwise it'd be `rest_authentication_required` or similar)
2. The request reached the SN custom permission_callback (`sn_rest_can_manage` from `inc/rest-api.php`, which requires `manage_options`)
3. That check rejected the call

So the password is probably **valid** but the bound user **lacks `manage_options`** — or the SN gate semantics changed and the password is still ok-as-credentials-but-not-as-authority. Three possibilities (rank ordered by likelihood):

1. **The App Password belongs to a non-admin user.** Easiest to verify: log into wp-admin → Users → look for `gha-deploy-plugin` (or whatever name was used) → check role.
2. **The App Password is for the right user, but that user's role was downgraded.** Same verification path.
3. **The SN auth gate `sn_rest_can_manage` was changed** to require something stricter than `manage_options` (e.g., a custom capability). Verify by reading `inc/rest-api.php` near the permission callback registration for `/purge-cache`.

This is different from the prior handoff's "secret may be stale" diagnosis (handoff item #5 of the previous handoff). Both are worth addressing in the QA pass; the response code points at #2 (user-perm issue), not #1 (stale credential).

## Process discipline outcomes for this arc

### What worked

- **Subagent-driven execution with fresh implementers per task** kept context clean across A → B → B-fix → C → D. No scope creep. No drift.
- **Two-stage review (spec then code quality) caught 4 important silent-failure bugs in Task B** that the tests passed despite:
  1. SQL ↔ PHP regex drift (`|in alpha` missing from SQL prefilter — silent coverage gap)
  2. `wp_json_encode` can return `false`; was unchecked → empty prompt → garbage AI response → silent post drop
  3. `max_tokens=600` truncates on high-candidate posts → mid-JSON parse fail → silent post drop (the v3.6.0 Task 2 lesson)
  4. Markdown-fence regex required both opening + closing fences; partial fences (no closer) failed → silent skip
- The re-review confirmed each patch via **out-of-band PHP one-liner verification** that the regression-guard test would actually fail on the OLD code and pass on the NEW. This is the right paranoia level — regression-guard tests that don't actually guard the regression are a known failure mode.

### What broke (worth keeping in mind)

- **The plan doc had three stale claims that the implementer correctly handled:**
  1. *"snt_ai_is_available and snt_ai_generate_with_constraints stubs already exist in this file"* — they did NOT; implementer added from scratch with `function_exists()` guards.
  2. *"$wpdb stub via the existing Stub_wpdb_* pattern"* — no such stub existed; implementer added an anonymous-class stub.
  3. *Predicted test count "44 prior + 12 new = 56"* was off by both inputs (actual: 46 + 16 = 62, then +2 from B-fixes = 64). Implementer trusted the harness output, not the plan.

  **Plan-doc bug-class lesson:** when plans cross task boundaries with predicted artifact counts (assertion totals, file line counts), the predictions decay fast. Always have the implementer report ACTUAL and update downstream artifacts (CHANGELOG, commit message totals) against actual, not predicted.

- **The plan's CHANGELOG block had two internal inconsistencies** (predicted total 425 vs. summed components 433; said "6 new tests" then listed 9 categories). The Task D implementer reconciled to actual: 441 total, 11 new test blocks. This is a Task-D-class job, not a planning failure — but worth flagging that plans-with-numbers should ship the math at execution time, not at planning time.
