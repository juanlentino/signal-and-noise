## Three primary changes

### 1. Login self-heal (fixes production /backend 404)

**Root cause** (diagnosed during brainstorm): the flush sentinel at `inc/login-hide.php:106-113` only re-flushed when `slug !== sentinel`. The sentinel could desync from the persisted `rewrite_rules` option (silent DB failure, plugin conflicts, WP's deferred-flush failure mode). Once desynced, the sentinel said "done flushing for backend" while `rewrite_rules` didn't actually contain `^backend/?$`. No self-healing path.

**Fix** (commit `c2cb0c5` + `6044af6`): two-part verify-before-trust pattern:
- Init priority 99 hook now also checks `isset($rules_option[$pattern])` — re-flushes when the rule is missing regardless of sentinel state.
- `save_login` handler force-deletes the sentinel on successful save (gated on `$ok`) — guarantees next-init flush fires.

**Tests**: `login-self-heal.php` (2 assertions, proves self-heal triggers on desync) + `login-self-heal-no-regression.php` (1 assertion, proves no DB write storm on happy path).

### 2. D-06 accessor enforcement

**Root cause**: 5 direct `get_option`/`update_option('sn_settings')` calls in `inc/admin-page.php` bypassed the `sn_setting()` per-request static cache. Latent bug (no co-request reads), one refactor from biting.

**Fix** (commits `f2e9b57`, `6044af6`, `0a7c25f`): new helpers `sn_setting_update()` + `sn_setting_reset_cache()` in `inc/settings.php`. The `sn_setting()` accessor was refactored to support a cache-reset sentinel path (`__sn_reset_cache__`). All 5 call sites in save_login + save_insights_settings replaced. Zero direct calls remain.

**Tests**: `sn-setting-update.php` (8 assertions) + `sn-setting-update-roundtrip.php` (3 assertions).

### 3. Audit-log retention controls (new feature)

**Why**: v3.8.3 hardcoded `SN_AUDIT_RETENTION_DAYS = 90`. v4.2.0 makes it configurable per `audit.retention_days` setting (7-365, default 90). Mirrors the inc/rss-plausible-tracker.php pattern.

**Fix** (commits `12e456e`, `80f1b31`): added `audit.retention_days` to `sn_settings_defaults()`, replaced 4 const refs in `inc/audit-log.php` with `sn_setting()` lookups, added UI form to Settings → Security → Audit log (number input 7-365), added `audit_save_retention` dispatcher branch in `inc/admin-page.php` with `max(7, min(365, $raw))` clamping.

**Tests**: `audit-retention-setting.php` (3 assertions) + `audit-retention-bounds.php` (8 assertions).

## Discipline lessons codified this cycle

This session added/reinforced these memory entries:

1. **The verify-before-trust pattern is now generalizable.** v4.1.4 (install-hook self-observation) → v4.1.5 (admin_init version-check) → v4.2.0 (login flush sentinel + D-06 accessor cache). Any cached sentinel must verify against underlying reality before being trusted. Worth a WORDPRESS-REFERENCE.md gotcha #39 if it bites again in v4.3.0+.
2. **Subagent-driven execution worked well.** 12 implementation subagents + per-task spec + quality reviews caught 2 real issues (Test 2 not actually proving cache-busting in Task 1; `delete_option` gating in Task 2) that would have shipped silently otherwise. Two-stage review is load-bearing, not ceremony.
3. **Implementer adaptations to plan inaccuracies are correct.** Task 9 implementer adapted the spec's invented `admin-post.php` flow to the actual codebase nonce pattern — the plan was wrong; the implementer was right. Trust judgment when it's grounded in source-read.
4. **Plan self-review caught real bugs.** The brainstorming-pass self-review caught that `inc/ai-drift-detect.php` doesn't exist (drift-detect prompt is in `inc/health-checks.php`); the plan-write self-review caught that D-06 actually has 5 call sites, not 3. Without these gates, both errors would have surfaced mid-execution as wasted work.
