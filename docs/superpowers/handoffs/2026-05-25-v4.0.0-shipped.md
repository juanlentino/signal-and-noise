## What's outstanding (for next session)

### 1. v4.0.x queue — inline-`<img>` alt Suggest + Copy (LIGHT)

The v4.0.0 scope explicitly deferred this. Inline-img findings (subject_type='inline_img' on the missing_alt check) currently render only the Edit button — no AI cell.

**The impl boundary needs settling:** inline imgs have no `attachment_id`; their `subject_id` is the parent post ID. Two options for v4.0.x:
- (a) Extend `snt_ai_alt_suggest_impl()` to accept polymorphic input (attachment_id OR (post_id + image_src) tuple)
- (b) Add a second impl + ability `signal-noise/ai-alt-inline-suggest` with separate schema

Apply itself stays deferred regardless — editing `post_content` to mutate inline `<img alt="…">` is a block-serialization-roundtrip problem (WORDPRESS-REFERENCE.md gotcha #4). v4.0.x ships Suggest + Copy only.

**Estimated:** ~70-100 LOC. Mostly a slightly different prompt + a new schema. Reuse the existing JS module pattern + admin UI cell-rendering — just need to detect inline-img findings and render a different cell variant (Suggest button → on-result render textarea + Copy button instead of Apply).

### 2. v4.1.0 queue — stale-posts + orphaned-media suggest+apply (FULL FEATURE)

The remaining 2 Health checks from the original v4.0.0 brainstorm scope. Both deferred because:
- **Stale posts (>12mo)**: AI suggests freshness updates as a read-only list (specific outdated statements + suggested rewrites). Apply is read-only / copy-to-editor because whole-post rewrites are risky for one-click apply. UX is "show me what's outdated"; user manually fixes.
- **Orphaned media**: AI keep/delete recommendation per attachment by analyzing context + filename. One-click Apply (delete attachment OR mark-kept).

**Pattern reference:** the canonical v4.0.0 suggest+apply pattern is captured in memory `feedback_health_suggest_apply_pattern.md`. v4.1.0 should follow it exactly — paired impl + REST + ability + extend the same shared JS module + extend `sn_health_render_suggest_cell()` to handle the new check types.

**Estimated:** ~200-250 LOC across 2 new impl files + 2 ability registrations + admin UI extensions + JS dispatch table additions.

### 3. GSC sitemap submission (USER-ACTION, NO CODE)

`/wp-sitemap.xml` was verified fetchable (HTTP 200, valid XML index, 4 sub-sitemaps) during Phase 1. User still needs to submit it in Google Search Console. ~15 minutes; no code change.
