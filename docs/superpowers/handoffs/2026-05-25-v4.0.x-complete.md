v4.1.0 is bigger than a single patch — it's the FIRST minor in the v4.x line and the original v4.0.0 brainstorm explicitly deferred multiple features to it. **This needs its own brainstorm.**

### Candidates surfaced across the v4.0.x arc

| Source | Candidate | Cost | LOC est. |
|---|---|---|---|
| Original v4.0.0 brainstorm | AI suggest for stale-posts (>12mo unmodified) — read-only suggestion list (whole-post rewrites too risky for one-click apply) | Medium AI (N calls per scan) | ~150-200 |
| Original v4.0.0 brainstorm | AI keep/delete recommendation for orphaned-media (one-click apply) | Medium AI | ~150-200 |
| v4.0.1 cost audit | Color drift Health check (palette from theme, regex scans post_content for hardcoded hex not in palette) | Zero AI (pure regex) | ~120 |
| v4.0.1 cost audit | Block pattern usage analytics (DB queries against post_content for pattern usage; feeds Insights recommendations) | Zero AI | ~80 |
| Original brainstorm | Cross-package E2E tests for the 4 filter contracts | Zero AI | ~150 |

### Key decisions for v4.1.0 brainstorm

1. **Combined or split ships?** All 5 candidates as one v4.1.0 (~700 LOC), OR split into v4.1.0 + v4.1.1 + v4.1.2 by feature?
2. **Which subset?** YAGNI says ship the highest-value 2-3 first; defer the rest to v4.1.x patches or v4.2.0.
3. **Stale-posts UX**: read-only list (deferred Apply) vs. partial-Apply (some rewrites are safe, like adding "Updated: 2026" notes)?
4. **Cost-aware ordering**: zero-AI candidates (color drift, pattern analytics, E2E tests) ship FIRST so we have buffer before the next AI-heavy feature?

### Cost ceiling reality check

Current production AI spend ~$4/month at v4.0.0 baseline. v4.0.1 cache reduced it ~80% on stable content (real number depends on Run-scan cadence). v4.0.2 + v4.0.3 added zero/microsecond cost. **v4.1.0 should be designed with the same cost-awareness lens** that informed the v4.0.x re-ranking — don't ship the conversational AI Copilot (B1 from original brainstorm) here; it remains the high-cost tier-4 candidate.
