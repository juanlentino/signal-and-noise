## Not-yet-verified (UI-interaction items)

The user has desktop-mode installed and active. Every wp-admin URL gets intercepted into desktop-mode's chrome-extension portal iframe. Claude_in_Chrome can read text + find elements + set form values inside that iframe, but **cannot fire input events, run JS, take screenshots, or click**, because the chrome-extension boundary blocks the Anthropic extension from poking another extension's content.

## Process discipline outcomes

The hard rule ([`feedback_skills_plugins_docs_always`](../../../.claude/projects/-Users-juanlentino-Projects-signal-and-noise/memory/feedback_skills_plugins_docs_always.md)) held this session. Every code change went through skill-driven workflow:

1. `superpowers:brainstorming` → 4 clarifying questions → 4 design decisions locked (scope: all events; tracking: universal DOING_CRON-gated; run-now: synchronous with 4 safety guards; AI exposure: read-only)
2. `superpowers:writing-plans` → 18-task plan with exact code per step
3. `superpowers:subagent-driven-development` → 15 implementer subagents + 15 spec-compliance reviewer subagents + 15 code-quality reviewer subagents + 1 final cross-cutting reviewer subagent + 1 blocker-fix implementer

**The final cross-cutting review earned its keep** — caught the `$valid_tabs` whitelist blocker that no per-task review could see in isolation. Task 10's spec/quality reviews both verified the page entry + dispatch case existed. Neither knew about the whitelist 200 lines away. Two-line bug → silent UI regression → caught after 15 isolated approvals. **This is the lesson worth carrying forward: when a feature requires changes at multiple coordinated points in a single file (entry list + dispatcher + whitelist + label map), the per-task review can verify each point but miss the COORDINATION constraint.**
