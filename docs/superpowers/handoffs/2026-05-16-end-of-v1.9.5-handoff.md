## Architectural invariants (preserve in future work)

1. **All form POSTs route through `sn_handle_admin_post()` in [`inc/admin-page.php`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/admin-page.php).** This is the central PRG (Post/Redirect/Get) handler hooked to `admin_init`. Each module's render callback is render-only — no inline `$_POST` processing. New form actions get added to the if/elseif chain in `sn_handle_admin_post`, plus a matching flash-code reader in the page-render code.

2. **Save status survives the redirect via `?sn_flash=<code>` query arg.** The handler appends it after `wp_safe_redirect`; the page-render reader translates it into a `$notices` entry.

3. **Single source of truth for admin pages: [`sn_admin_pages()`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/inc/admin-page.php) array.** Each entry has `slug`, `tab`, `label`, `title`, `subtitle`. Registration loop reads this; URL dispatcher reads this; subtitle lookup reads this. Don't drift between the array and any hardcoded list.

4. **Hook suffix collection** filters out `false` returns from `add_submenu_page` (gotcha #15). The enqueue guard uses `in_array($hook, sn_admin_page_hooks(), true)`.

5. **Design system classes live in [`assets/admin.css`](https://github.com/juanlentino/signal-and-noise-tools/blob/main/assets/admin.css)** with CSS vars (`--sn-surface`, `--sn-border`, `--sn-text-muted`, `--sn-space-*`, status colors). Don't reintroduce inline styles even for "just this one thing" — they compound.

## Knowledge captured in the theme repo

WORDPRESS-REFERENCE.md gained 6 gotchas this session from actual WP core source reading + Plugin Handbook audits:

- **#14** — `add_submenu_page` auto-prepends a duplicate-parent submenu; escape hatch is matching the parent slug
- **#15** — `add_menu_page` returns hook unconditionally; `add_submenu_page` returns `false` for low-cap users (asymmetry)
- **#16** — Neither function deduplicates slugs (silent phantom entries)
- **#17** — `load-{hookname}` hook is the WP-recommended form-processing location (allows PRG)
- **#18** — Plugin Handbook recommends Settings API; custom forms are defensible deviation but mean owning what the API does for you
- **#19** — Custom settings pages with inline `$_POST` need PRG explicitly (Settings API does it for you)
