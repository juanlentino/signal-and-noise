### Item 2: Detector cleanup (3 sub-items)

**File:** `inc/pattern-adoption-detect.php`

**Fix:** Change line 179 from `if ( '' !== $cite_html ) {` to `if ( '' !== trim( strip_tags( $cite_html ) ) ) {`. Also add a test fixture for this case:

```php

_tas_post( 211, array( $quote_empty_cite ) );
$fp = md5( serialize_block( $quote_empty_cite ) );

ps_true( false === strpos( $result['suggestion_markup'], 'sn-pull-quote__attribution' ), 'Test 11.1: attribution omitted when cite has only empty inline tags' );
```

### Item 6: Abilities wrapper `function_exists` guard

**File:** `inc/abilities-ai-pattern-adoption.php` (lines 128 + 144 — the two wrapper functions).

**Issue:** All 7 health-ability wrappers in `inc/abilities-ai-health.php` (lines 311, 324, 340, 358, 378, 394, 407) use this defensive pattern:

```php
if ( ! function_exists( 'snt_ai_X_impl' ) ) {
    return new WP_Error( 'snt_helper_unavailable', '...', array( 'status' => 500 ) );
}
```

The pattern-adoption wrappers go straight to the impl call. The `require_once` chain protects against the failure mode in production (impl files always loaded before abilities), but consistency with canonical template diverges.

**Fix:** Add the guard to both wrappers:

```php
function snt_ability_pattern_adoption_suggest( $input ) {
    if ( ! function_exists( 'snt_ai_pattern_adoption_suggest_impl' ) ) {
        return new WP_Error( 'snt_helper_unavailable', __( 'Pattern-adoption suggest helper not loaded.', 'signal-noise-tools' ), array( 'status' => 500 ) );
    }
    return snt_ai_pattern_adoption_suggest_impl(
        (int)    ( $input['post_id'] ?? 0 ),
        (string) ( $input['block_fingerprint'] ?? '' ),
        (string) ( $input['pattern_type'] ?? '' )
    );
}
```

Same shape for `snt_ability_pattern_adoption_apply`. ~10 LOC total (5 per wrapper).

### The 4 plan-vs-reality drift lessons (THE meta-takeaway)

This session caught 4 plan-vs-reality drifts at code review, all same root cause: **plan was written from external assumptions about how project subsystems work, without reading the actual code.**

1. **v4.3.0 Task 2 (CRITICAL):** Plan specified `<!-- wp:signal-noise/pull-quote -->` markup. Reality: pattern slug ≠ registered block type. Verifiable by reading `theme/patterns/pull-quote.php`.
2. **v4.3.0 Task 4 (CRITICAL):** Plan specified `add_action('admin_post_$slug', ...)` hook. Reality: project uses `sn_handle_admin_post()` dispatcher. Verifiable by reading `inc/admin-page.php`.
3. **v4.3.0 Task 5 (Minor):** Plan assumed wrapped `{slug, args}` shape for `$GLOBALS['__test_registered_abilities']`. Reality: flat keying with slug as array key.
4. **v4.3.0 Task 6 (Minor):** Plan showed sibling `document.addEventListener('click', ...)`. Reality: project uses single dispatcher with `if/else if` branches inside `init()`.

**Generalization (now in memory):** for any task that wires into an existing project subsystem (admin pages, REST routing, JS event handlers, test stubs, registries), plans should embed an explicit instruction: "Before writing this task's code, read the existing implementation of `<dispatcher / hook / dispatch table>` at `<file:line>` and mirror its convention."
