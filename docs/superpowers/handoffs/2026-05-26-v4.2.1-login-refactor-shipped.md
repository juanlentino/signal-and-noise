# Handoff — 2026-05-26 (plugin v4.2.1 — login refactor shipped)

**Why this exists:** v4.2.0 shipped a verify-before-trust self-heal for the production `/backend` 404 bug, but `/backend` continued to 404 after install. Investigation revealed v4.2.0's self-heal addressed *symptoms* (sentinel desync) rather than the *root cause* — the entire `add_rewrite_rule` routing approach used since v1.5.0 was architecturally wrong. The proven peer implementation we replaced (`wps-hide-login`) doesn't use rewrite rules at all. v4.2.1 refactors `inc/login-hide.php` to mirror `wps-hide-login`'s `plugins_loaded` intercept pattern.

## What went wrong with v4.2.0

| Symptom | What v4.2.0 thought was the cause | What was actually the cause |
|---|---|---|
| `/backend` returns 404 in production | Flush sentinel desync from `rewrite_rules` option | The `add_rewrite_rule` routing path is fundamentally fragile — depends on a chain of 4 assumptions any of which can fail |
| Self-heal at init priority 99 didn't fix it | Self-heal needs ONE init request to fire after install | Even after firing, the rule may not persist (deferred-flush failure, Cloudways Nginx routing, other plugins, CDN) |
| Manual `Settings → Permalinks → Save Changes` didn't fix it | Cloudflare/Breeze cache | Layer-deeper routing issue — the rewrite engine itself wasn't routing `/backend` to wp-login.php |

The breakthrough came from the user's question: **"We'd research how wps-login-hide does it, maybe? The plugin I had before this."**

This is exactly Phase 2 of `superpowers:systematic-debugging`: "Find Working Examples → Compare Against References." A 15-minute read of `wps-hide-login.php` (511 LOC, single file at [WPPlugins/wps-hide-login](https://github.com/WPPlugins/wps-hide-login)) revealed the structural mistake: **wps-hide-login does NOT use `add_rewrite_rule`**. It intercepts at `plugins_loaded` priority 2 and does `$pagenow = 'wp-login.php'` directly — no rewrite engine involvement at all.

## What v4.2.1 ships

### Removed (rewrite-engine routing path)

- `add_rewrite_rule()` registration (was at init priority 10).
- Init priority 99 flush sentinel + verify-before-trust self-heal (v4.2.0's symptomatic fix — no longer needed without the rewrite rule).
- `delete_option('sn_login_rewrites_flushed')` force-flush call in `save_login` handler.
- `tests/login-self-heal.php` + `tests/login-self-heal-no-regression.php` (tested removed code).

### Added (plugins_loaded intercept path)

- `sn_login_intercept_request()` at `plugins_loaded` priority 2:
  - Returns early for allowlisted endpoints (admin-ajax, async-upload, wp-cron, /wp-json/, /feed).
  - Sets `$GLOBALS['sn_login_block_wp_login'] = true` if REQUEST_URI hits `/wp-login.php` directly.
  - Sets `$GLOBALS['sn_login_serve_form'] = true` + `global $pagenow = 'wp-login.php'` if REQUEST_URI path equals `/<custom-slug>`.
- `sn_login_handle_request()` at `wp_loaded`:
  - Branch 1 (serve-form flag): `require_once ABSPATH . 'wp-login.php'; die;` — exactly what `wps-hide-login` does.
  - Branch 2 (block_wp_login flag): audit-log counter + status_header(404) + 404 template + exit.
  - Branch 3 (/wp-admin + unauthenticated): audit-log counter + status_header(404) + 404 template + exit.
- Preflight check for `rename-wp-login` (parity with `wps-hide-login`'s own conflict-detection at [wps-hide-login.php:125-141](https://github.com/WPPlugins/wps-hide-login/blob/master/wps-hide-login.php#L125)).
- `tests/login-intercept.php` (19 assertions): serve-form for matching slug + 4 variants (trailing slash, query string, etc.) + block-wp-login for direct access + 5 allowlist tests + substring-match prevention + nested-path rejection + action registration priority verification.

### Preserved (all v1.5.0–v4.2.0 hardening, unchanged)

- `SN_LOGIN_BYPASS` constant — emergency escape hatch restores `/wp-login.php` immediately.
- `SN_LOGIN_SLUG` constant — wp-config override for per-environment slugs.
- `wps-hide-login` v2.1.1 preflight (require BOTH `is_plugin_active` AND `file_exists`).
- URL filters: `site_url`, `network_site_url`, `wp_redirect` — replace `/wp-login.php` in generated URLs.
- Audit-log counter integration: `wp_login_404` and `wp_admin_unauth_404` still fire for blocked requests.
- Allowlist: admin-ajax / async-upload / wp-cron / /wp-json/ / /feed never get touched.

1. **NEW: [feedback_read_peer_implementation_first.md](../../../../../../.claude/projects/-Users-juanlentino-Projects-signal-and-noise/memory/feedback_read_peer_implementation_first.md)** — When fixing a fragile component that REPLACES or runs PARALLEL to a proven third-party impl, read that impl's source FIRST. Generalizes the read-framework-source rule to peer plugins.

2. **REINFORCED: [feedback_read_framework_source.md](../../../../../../.claude/projects/-Users-juanlentino-Projects-signal-and-noise/memory/feedback_read_framework_source.md)** — 3rd reinforcement now (v3.7.1, v3.8.x theme abilities, v4.2.x login). The pattern is consistent: every time we ship code that depends on a framework primitive (WP rewrite engine, AI client `__call`, theme abilities) without reading the primitive's source, we ship a bug.

The trigger fired exactly when it should have. The user's suggestion to "research how wps-login-hide does it" was the architectural pivot.
