## Plugin v3.0.2 — what landed

Polish patch addressing Task 9 + Task 12 reviewer notes from the v3.0.0 session. The cross-cutting reviewer's lesson ("a feature requiring changes at multiple coordinated points in a single file can pass per-task review and still miss the COORDINATION constraint") is now encoded architecturally.

### Tab registry SSOT

`sn_admin_pages()` is now the only place an admin tab is registered. Two derived helpers replace the previously-duplicated inline arrays:

```php
function sn_admin_page_valid_tabs() {
    return array_column( sn_admin_pages(), 'tab' );
}

function sn_admin_page_tab_labels() {
    return array_column( sn_admin_pages(), 'label', 'tab' );
}
```

`sn_theme_options_page()` now calls those instead of holding its own copies. Adding a future tab = one entry in `sn_admin_pages()`; whitelist, label map, dispatch list, hook enqueue all derive from it automatically.

### A11y on the Cron dashboard renderer

- `<label for="sn-cron-filter">` (`.screen-reader-text`) — the filter input is now programmatically labeled
- `<caption class="screen-reader-text">` on the table
- `<th scope="col">` on column headers + `<th scope="row">` on the hook cell
- SN / orphan badges have `title` attributes + `<span class="screen-reader-text">` prefixes ("Signal and Noise owned:" / "Warning:")
- Run-now buttons get `aria-label="Run cron event {hook} now"` so screen readers disambiguate identically-labeled buttons across rows
- Plural-aware count line via `_n()` + `number_format_i18n()`

### i18n on the JS layer

`wp_localize_script` now passes a `sntCronI18n` bag with every user-facing string. JS consumes them with defensive English fallbacks so a botched deploy that loses the localize call still leaves the module functional in degraded English mode.

## Theme deploy.yml fix — what changed

The v9.0.0 deploy at 23:46Z flagged red because the **CF purge step returned HTTP 401**. Root cause: this repo's `WP_DEPLOY_APP_PASSWORD` GH secret was revoked back in Phase 13 (2026-05-16) and never rotated. The plugin repo's same-named secret IS valid — that's why plugin v3.0.2 just purged cleanly. They're scoped per-repo.

Patch ([commit 4ec38b6](https://github.com/juanlentino/signal-and-noise/commit/4ec38b6)):

1. **`continue-on-error: true`** on the purge step. Files are live the moment git-pull lands; CF edge revalidates naturally via the new `last-modified` header on every theme asset. A missed purge is a latency artifact, not a deploy failure.
2. **Empty-secret skip + targeted 401/403 diagnostics.** The step still attempts the purge when the secret IS valid, and still emits `exit 1` + a `::warning::` annotation when it can't auth — but the overall run stays green. The warning lines spell out the exact `gh secret set` command for remediation.
3. **Re-validated** ([run 26197226420](https://github.com/juanlentino/signal-and-noise/actions/runs/26197226420)) — same v9.0.0 codepath, same stale-secret 401, but now the run is green ✓ with a yellow ⚠️ on the purge step.

## Orphan cron cleanup — what shipped + what got cleaned

### The tool

New `cron-orphan-cleanup.yml` workflow in the plugin repo. Reusable, not a one-off — as plugins come and go on the site, orphans will accumulate again. Two safety properties:

- **Default is DRY-RUN.** Must explicitly pass `-f dry_run=false` to delete.
- **Defense in depth:** remote-side `wp eval "echo has_action('$hook') ? '1' : '0';"` guard refuses to delete any hook with a live handler, even if someone supplies it on the optional `hooks` input list.

Inputs are typed (boolean / string) and flow via `env:` into the heredoc body as positional args, so the runner shell never expands user-controlled values into a `run:` command.

Trigger:
```bash
gh workflow run cron-orphan-cleanup.yml --repo juanlentino/signal-and-noise-tools \
  --ref main -f dry_run=true
# then, after reviewing the WOULD lines:
gh workflow run cron-orphan-cleanup.yml --repo juanlentino/signal-and-noise-tools \
  --ref main -f dry_run=false
```

## Lessons captured / encoded

| Lesson | Where it lives now |
|---|---|
| `sn_admin_pages()` is the only place to register an admin tab | Encoded architecturally via `sn_admin_page_valid_tabs()` + `sn_admin_page_tab_labels()` helpers + 135-assertion regression guard |

| WP-Admin app passwords scope per-repo, not per-org | Now documented in deploy.yml comment + the remediation `::warning::` message |
