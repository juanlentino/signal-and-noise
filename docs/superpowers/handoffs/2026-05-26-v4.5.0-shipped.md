## Section 3: What's in the Block Migration tool

### File structure (plugin repo)

```
inc/block-migrations-detect.php     — Walks all published posts; flags h3-without-preceding-h2.
                                       Fingerprint = md5(serialize_block($node)). Caches scan
                                       in user transient at HOUR_IN_SECONDS.
inc/block-migrations-suggest.php    — Deterministic h3 → h2 transformation: unset attrs.level
                                       + rewrite innerContent + innerHTML. No AI.
inc/block-migrations-apply.php      — Fingerprint-validated swap: parse_blocks → walk →
                                       md5 match → in-place swap → serialize_blocks →
                                       wp_update_post. 409 conflict on mismatch.
inc/block-migrations-admin.php      — Tools sub-tab section renderer + dismiss REST endpoint.
                                       Hooks into sn_admin_block_migrations_tab.
inc/abilities-block-migrations.php  — 4 ability registrations under category 'tools':
                                       scan / suggest / apply / dismiss.
```

### REST surfaces

```
POST /signal-noise/v1/tools/block-migrations-scan       (manage_options)
POST /signal-noise/v1/tools/block-migrations-suggest    (edit_post)
POST /signal-noise/v1/tools/block-migrations-apply      (edit_post)
POST /signal-noise/v1/tools/block-migrations-dismiss    (edit_post)
```

## Section 7: Lessons captured this session

1. **Source-verify before plan-writing.** The brainstorm spec's WS2 ("JS-client flip: ability-first, REST-fallback") was based on a stale premise. Reading `assets/ai-*.js` BEFORE drafting tasks against them revealed the flip already shipped in v2.5.0. Killing WS2 saved ~100 LOC of phantom work and reduced the plugin scope from 3 workstreams to 2. The discipline trail is preserved in the spec's §2 revision note.

2. **Two-stage review catches what one-stage misses.** The spec-compliance review caught a deviation in Task 4 (dead-code branch from a malformed test fixture — the implementer extended the impl instead of fixing the test). The code-quality review caught Critical and Important issues in Tasks 3 + 6 (regex word-boundary bug, missing flash notice handler, hook-name convention deviation). Neither would have surfaced via just one review layer.

3. **Cleanup cycles are signal, not noise.** 5 of the 15 commits in the cycle were post-task cleanups driven by review feedback. Each captured a real defect or coverage gap. The pattern: implementer ships → reviewer flags → fixer cleans up → reviewer re-approves. This is the discipline working, not the discipline being inefficient.

4. **Match project convention, not spec literal text.** The spec said `sn_admin_tools_tab` (generic). The project convention is per-sub-tab flat hooks (`sn_admin_cloudflare_tab`, `sn_admin_health_tab`). The implementation diverged from the spec's literal text to match the project pattern. Documented at commit `e80c1f2` (the cleanup that renamed `sn_admin_tools_block_migrations_tab` → `sn_admin_block_migrations_tab`).

5. **Test assertion counts in plans drift from reality.** The plan said: detect=14, suggest=12, apply=12. Actual: 13 → 15 (cleanup +2), 14 → 18 (cleanup +4), 12 → 14 (cleanup +2). Plan totals expected 931; actual 940. The drift came from miscounting at plan-write time PLUS legitimate cleanup additions. Don't treat plan numbers as gospel — verify against the actual `php tests/*.php` output.

6. **PHP `\b` word-boundary doesn't do what you'd expect.** `/<h3(\b[^>]*)>/` matches `<h3-custom>` because `-` is a non-word character and `\b` fires between `3` and `-`. Use explicit lookahead `(?=[\s>])` for tag-matching. Caught at Task 3 code-quality review; fix at commit `cf8a571`.

7. **`(array)` cast is sufficient — no need for `is_array` followup.** PHP's `(array)` cast handles `false`, `null`, `""`, scalars all correctly. The `if (!is_array($x)) { $x = array(); }` follow-up is dead code. Caught at Task 5 code-quality review; fix at commit `0b21a12`.
