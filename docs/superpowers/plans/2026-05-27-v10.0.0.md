>
> **TENTATIVE STATUS:** Written ahead of the v10.0.0 BC for visibility. Some sections — Task 2 (drop pre-7.0 compat code) and Task 4 (conditional theme.json v4 migration) — depend on the LIVE state at BC time. At v10.0.0 BC time (after v5.0.0 + v9.6.x gate stable), this plan must be refreshed.

**Goal:** Ship the theme-side breaking changes the v9.6.0 prep minor announced — HARD-raise WP 7.0 baseline, drop pre-7.0 compat code, promote `@deprecated` PHPdoc to runtime warnings (if any v9.6.x-era deprecations exist), conditionally migrate theme.json schema v3 → v4 if WP ships v4 by then.

**Architecture:** Smallest plan in the cycle — theme is already mostly WP 7.0-ready (per `2026-05-27-v10.0.0-scope.md` finding: 79 KEEP / 0 RENAME / 0 REMOVE / 1 SCHEMA-CHANGE-conditional). v10.0.0 ships mostly as a POLICY major (the WP floor raise) with minimal code change.

**Tech Stack:** PHP 8.0+, WordPress 7.0+ (HARD requirement), Abilities API, no new JS, no new CSS, theme.json schema.

**Reference spec:** [`docs/superpowers/specs/2026-05-27-v5-and-v10-paired-cycle-design.md`](../specs/2026-05-27-v5-and-v10-paired-cycle-design.md) — §6 v10.0.0 major scope.

**Sequencing prerequisite (HARD GATE):**
1. Plugin v5.0.0 shipped + UATed stable
2. Theme v9.6.0 + v9.6.x patches gated stable
3. v10.0.0 BC convened, this plan refreshed against live state + plugin v5.0.0 actuals

## Cross-package coordination

Per spec §7:
- **v10.0.0 BC consumes plugin v5.0.0 (or v5.0.x) state.** Verify the plugin's 36 abilities are still live (no rename, etc.).
- **Paired ship.** v5.0.0 + v10.0.0 ship in the SAME session if possible. Install order: typically plugin first (it's the dependency for the cross-package contracts), then theme. WP 7.0 floor on both means installing one then the other on a WP < 7.0 environment would fail at install-time on the first one — the user must upgrade WP first.
